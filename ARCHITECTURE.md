# poker trainer ARCHITECTURE (V1.0)

## Notes — Home Screen, Mode Selection, and Game Entry Flow

The application uses three primary screens in a navigation hierarchy: the Root screen, the Mode Selection screen, and the Game screen. Buttons morph between these screens rather than instantly appearing or disappearing, which creates visual continuity and tells the user which screen they are on through layout alone.

### Root Screen Layout

The root screen has three layout regions: top-left, middle, and top-right.

- **Top-left region:** application logo (small, anchored to top-left corner).
- **Middle region:** a 2x2 grid of four buttons. Position-by-position: Play occupies the top-left of the grid, Settings occupies the top-right of the grid, Shop occupies the bottom-left of the grid, and Help occupies the bottom-right of the grid. "Top-left of the grid" refers to the grid's internal layout, not the screen's top-left corner — the entire grid sits in the screen's middle region.
- **Top-right region:** a single small Home icon. On the root screen, clicking the Home icon reloads the page (it has no other navigational purpose since the user is already at the root). Root screen focus list, registered with focus_manager on screen entry: Play → Settings → Shop → Help → Home, wrapping from Home back to Play. Default focus pointer on screen entry is 0 (Play). Tab advances through this list; shift-tab reverses. Enter on a focused element activates it (Play opens the morph to Mode Selection, Settings/Shop/Help open their respective modals, Home reloads the page).

### Transition From Root to Mode Selection

When the user clicks Play on the root screen, all four middle-grid buttons begin a morph animation that takes them to new positions on the Mode Selection screen. The morph is eased (using a smooth easing curve, not linear) and staggered by approximately 50 milliseconds per button so the four motions do not happen simultaneously. During this morph, the Play button slides from its position in the middle 2x2 grid to the top-left region of the screen. As it slides, its visual label transforms from "PLAY" into "STANDARD." After the morph completes, the button reads STANDARD and is anchored in the top-left region. The Settings, Shop, and Help buttons each slide from their positions in the middle 2x2 grid to the top-right region of the screen. As they slide, their visual representations transform from full-size buttons into small icons. They arrange themselves into a single horizontal row in the top-right region, positioned to the left of the Home icon (which has remained in place throughout, anchoring the cluster). After the morph completes, the top-right region contains a horizontal row of four icons. From left to right, these are: Shop (storefront icon), Help (question mark icon), Settings (cog icon), and Home (the icon that was always there). The Home icon's position has not changed; the other three icons have arrived to its left.

### Mode Selection Screen Layout

After the morph completes, the screen has the following layout:

- **Top-left region:** the STANDARD button (which was Play before the morph).
- **Middle region:** three new buttons appear, arranged horizontally and centered vertically in the screen. From left to right: Aggressor, Caller, Custom.
- **Top-right region:** the four-icon cluster (Shop, Help, Settings, Home). The user now has four ways to launch a game from this screen: clicking STANDARD in the top-left, clicking Aggressor in the middle, clicking Caller in the middle, or clicking Custom in the middle (which opens an intermediate modal popup before launch). Mode Selection focus list, registered with focus_manager on screen entry: STANDARD → Aggressor → Caller → Custom → Shop → Help → Settings → Home, wrapping from Home back to STANDARD. Default focus pointer on screen entry is 0 (STANDARD). Tab advances through this list; shift-tab reverses. Enter on a focused element activates it (STANDARD/Aggressor/Caller launch their respective game modes, Custom opens the configuration popup, cluster icons open their respective modals, Home returns to Root).

### Behavior of the Four Game-Launch Options

STANDARD button (top-left): clicking this immediately launches the game in Standard mode, which uses a 50/50 weighted distribution between Aggressor scenarios and Caller scenarios. Aggressor button (middle): clicking this immediately launches the game in Aggressor mode (100% Aggressor scenarios). Caller button (middle): clicking this immediately launches the game in Caller mode (100% Caller scenarios). Custom button (middle): clicking this does NOT launch a game immediately. Instead, it opens a centered modal popup that allows the user to configure custom weight percentages between Aggressor and Caller scenarios. The user must complete configuration in the popup and click the popup's Play button to launch the game.

### Custom Mode Configuration Popup

The Custom popup is a centered modal overlay. It can be dismissed without launching a game by clicking its X close button in the top-right of the popup, or by clicking outside the popup. The popup's contents, from top to bottom: A row labeled "Aggressor" on the left side. To the right of the label is a numeric text input field, followed by a percent (%) symbol. The text input accepts only integer values in the range 0 through 100. Directly below this row is a slider bar that spans the full width of the modal. The slider represents the Aggressor weight value. A row labeled "Caller" on the left side, structured identically to the Aggressor row: numeric text input on the right with a percent symbol. Directly below this row is a slider bar that spans the full width of the modal, representing the Caller weight value. At the bottom of the popup is a horizontal row of three action buttons: Save (left), Reset (middle), Play (right).

### Coupled Input Behavior

The Aggressor weight and Caller weight always sum to exactly 100. The four input methods (Aggressor slider, Aggressor text field, Caller slider, Caller text field) are all coupled to this constraint. When the user changes any one of them, the corresponding opposite input automatically adjusts so the sum remains 100. For example, if the user drags the Aggressor slider to 70, the Caller slider automatically moves to 30, and the Caller text field updates to display 30. If the user types 25 in the Caller text field, the Aggressor text field updates to 75 and both sliders adjust accordingly.

### Action Button Semantics

The three action buttons (Save, Reset, Play) have distinct, non-overlapping functions. No button infers another button's behavior — the user must explicitly click each action they want to perform. Save: Persists the current weight configuration to local storage (via the IDBFS persistence layer specified in Module 7). After clicking Save, the next time the user opens the Custom popup, the saved weights pre-populate. Save does NOT launch the game and does NOT close the popup. Reset: Restores the weight configuration to whatever was last saved. If no Save has ever been performed, Reset returns the weights to a default of 50/50. Reset does NOT launch the game and does NOT close the popup. Play: Launches the game immediately using the current weight configuration shown in the popup. Play does NOT persist the weights. If the user has adjusted the weights since the last Save and clicks Play without first clicking Save, the next time they open the Custom popup, the weights will revert to the last-saved values (or to 50/50 if no save exists). This explicit separation means a user who configures custom weights and clicks Play will play with those weights for that session only. A user who wants those weights to persist across sessions must click Save before Play. Custom Mode Configuration popup focus list, registered with focus_manager on popup open: Aggressor text input → Aggressor slider → Caller text input → Caller slider → Save → Reset → Play → X close, wrapping from X close back to the Aggressor text input. Default focus pointer on popup open is 0 (Aggressor text input). Tab advances through this list; shift-tab reverses. The sliders are coupled to the text inputs per existing spec; modifying the slider value via arrow keys (see Notes — Keyboard Focus Behavior) propagates to the corresponding text input and to the opposite Aggressor/Caller pair to maintain the sum-100 constraint. Within the Aggressor text input and Caller text input, up arrow / right arrow increment the value by 1, down arrow / left arrow decrement by 1, with hold-to-repeat at standard text editor cadence (initial delay ~500ms, repeat ~50ms). Values clamp at 0 and 100. Increment/decrement automatically propagates to the coupled opposite input. Within the Aggressor slider and Caller slider (single-handle range inputs), left/down arrow decrement the slider value by 1, right/up arrow increment by 1, with the same hold-to-repeat cadence. The slider value clamps at 0 and 100.

### Transition From Mode Selection to Game Screen

Once any of the four launch paths is triggered (clicking STANDARD, Aggressor, Caller, or Play within the Custom popup), the screen transitions into the Game screen. During this transition the STANDARD button in the top-left disappears (slides off-screen, fades, or otherwise transitions out — specific motion technique determined during implementation). The middle region's three buttons (Aggressor, Caller, Custom) disappear and are replaced by the game itself. The Home icon in the top-right cluster transforms in place into an X icon — same screen position, different glyph. Implementation can use a cross-fade, rotation, or other icon-swap transition. Shop, Help, and Settings icons remain unchanged in their positions.

### Game Screen Layout (Final)

The game screen layout is locked as follows:

- **Top-left:** chip denomination legend (top), then pot size, then blinds (bottom). The denomination legend is a horizontal row of chip PNGs representing all denominations in the active denomination set, with each chip's dollar value labeled directly beneath the chip in legend text. Layout per chip slot: chip PNG on top, dollar value label below. Chips are arranged left-to-right in ascending denomination order (smallest leftmost). The legend's column count varies based on the active denomination set — Micro tier shows 4 chips, Mid tier shows 4 chips, High tier shows 5 chips, Nosebleed tier shows 5 chips, Fixed mode shows 5 chips. Pot size and blinds render below the legend with a small vertical gap separating them.
- **Left-middle:** math input boxes with keybinds.
- **Top-middle:** community cards, fanned spread, face-up if dealt.
- **Bottom-middle:** hole cards, fanned spread, face-up.
- **Middle (the table):** opponent positions arranged around the table edges with chip stacks at each seat. In Caller scenarios, the active opponent's pushed-forward chips + floating bet amount sit between their seat and the pot. Position indicators (UTG, HJ, CO, BTN, SB, BB) live at each seat on the table. The pot itself sits at the center of the table. Opponent stack sizes are rendered as numeric values at their seats — the dollar amount (Cash mode) or big blind multiple (BB mode), per the Global Unit Toggle. The number IS the visual representation of each opponent (no chip visualization, no separate avatars).
- **Right side:** dealer rendered in profile (side view, not facing the camera). Top-right of the right side is the persistent cluster (Shop, Help, Settings, X), with the Visual Countdown timer rendered directly below the cluster, right-aligned to the cluster's right edge with a small vertical gap separating them. Game screen focus list, registered with focus_manager on screen entry: math input 1 → math input 2 → (additional math inputs in the order they spawn per scenario type) → bet size group (single tab stop, only present in Aggressor scenarios with the Bet Sizing Engine setting enabled) → Shop → Help → Settings → X, wrapping from X back to math input 1. Default focus pointer on screen entry is 0 (first math input). The bet size group is a single tab stop containing the four bet size buttons (1/3 Pot, 1/2 Pot, Full Pot, Overbet). When the group has focus, the focus indicator renders as a 2px outline using border_focus around the bounding box of the entire button row, not around any individual button. Keys 1-4 select the corresponding button within the group per existing Module 5 spec; the global 1-6 keybinds for math input focus are suppressed while the group has focus. Tab moves focus out of the group to the next element (Shop); shift-tab moves out backward to the previous math input. Mouse click on any of the four buttons selects that button, snaps the focus pointer to the group, and activates keyboard mode if not already active. The dealer is not included in the focus list; the dealer remains mouse-only for the 22-click easter egg interaction. Chip Stack Rendering Convention: Chip stacks rendered as denomination-accurate columns appear in three places: bet chips pushed forward in Caller scenarios, the main pot, and the side pot when present. Opponent seats display only a numeric stack value, not chip columns. The denomination-sorting convention below applies only to the bet, main pot, and side pot. Each stack location is a cluster of single-denomination columns rather than a single mixed column. Within each cluster, columns are arranged left-to-right in descending denomination order (highest-value chips on the left, lowest on the right). Each column's height scales linearly with the count of chips in that denomination using a fixed pixels-per-chip value. Examples: A player with $1000 in pure $100 chips renders as a single tall column of 10 black chips. A player with $437 (using the Mid-tier denomination set) renders as four columns side by side: 4 black ($400), 1 green ($25), 2 red ($10), 2 white ($2). A short-stacked player with $40 renders as a single short column of 8 red chips. A $50 bet (1 green + 5 red) pushes forward as two columns: 1 green chip and a stack of 5 red chips, in descending-denomination order. All-In Visualization: When a side-pot scenario triggers, the all-in opponent's chip area is empty (no columns rendered) since their entire stack is now in the pot. Their seat retains a position label and may be marked with a colored ring to identify them as the side-pot source. Pot Composition: The main pot's chip cluster is constructed by greedy-merging all bets that contributed to it into denomination columns. The side pot is a separate, slightly offset cluster, similarly merged from the all-in player's matched contributions only. Chip-Counting Drill (Show/Hide HUD): When numeric labels are hidden, the user counts chips by reading column heights and multiplying by the column's denomination color. The fixed pixels-per-chip rendering ensures stacks are visually countable. Sorted-denomination columns make this calibration directly transferable to real-world chip reading, since real cardrooms enforce the same sorting convention. When HUD is hidden, opponent stack numbers also hide. The user has no chip visualization to fall back on for opponent stacks. This is intentional: the math drill does not depend on opponent stack precision, and hiding the numbers reinforces that the active surfaces (bet, pot) are where the user's attention belongs. Denomination Legend Behavior: The chip denomination legend in the top-left region serves as a visual reference key, displaying which chip colors map to which dollar values for the active scenario. The legend updates per scenario when the Chip denomination mode setting is "Stake-scaled" — each new scenario's blind level determines the active tier, and the legend re-renders to show that tier's denomination set. When the Chip denomination mode setting is "Fixed," the legend is constant across all scenarios, always showing the same 5-chip denomination set ($1, $5, $25, $100, $500). The legend remains visible regardless of the Show/Hide HUD setting. Chip faces do not display dollar values prominently — chips are identified by color, not by reading numerals — so the legend is the user's primary reference for which color represents which denomination. The HUD-hidden chip-counting drill removes floating numeric labels (call amounts, pot total when hidden) to force the user to count chips manually, but the legend stays present because it is a reference key, not a calculation aid. The drill trains visual estimation (count column heights, multiply by denomination, sum), which is the transferable real-cardroom skill. The legend updates instantly when a new scenario spawns. There is no fade or animation; the new chips appear at scenario load alongside the rest of the table state.

### Side Pot Visualization

When the engine triggers an All-In side pot scenario, the table renders two visually distinct pot piles in the center: the main pot and the side pot, slightly offset from each other so the user immediately registers two separate pools of chips rather than one. The all-in opponent who triggered the side pot is rendered with a visual marker on their seat — depleted chip stack, a colored ring around their position, or an "ALL-IN" indicator — communicating that this player is the source of the side pot split. Together, the two pots and the all-in marker tell the side-pot story without text labels, consistent with the application's design principle of communicating game state through visual table cues rather than explicit text. The Side Pot Engine triggers an All-In side pot scenario approximately 10% of the time. This frequency is configurable via the Settings panel.

### HUD Visibility Independence

HUD element visibility (controlled via the Show/hide HUD toggle) is independent of countdown timer visibility (controlled via the Show countdown timer toggle). The time penalty applies based on actual elapsed time regardless of either toggle's state. Users configure each setting independently based on which training mode they want.

### Scenario State Communication on the Game Screen

The application does not use explicit text labels to inform the user whether the current scenario is a Caller scenario or an Aggressor scenario. The user infers scenario type from the visual state of the table itself, mirroring how poker is read in real life. In Caller scenarios, the opponent has already acted and placed a bet. This is rendered visually as physical poker chips pushed forward from the opponent's seat position toward the pot. A numeric value indicating the bet size (for example, "$50") floats in the immediate vicinity of the chip stack — adjacent to or above the chips — so the user can read the call amount without having to count chip denominations. In Aggressor scenarios, the opponent has not acted. The chip area in front of the opponent is empty. The pot in the middle of the table reflects only the existing contribution from blinds and antes; no additional chips have been pushed forward. The combination of table state (chips pushed forward or not) and the math inputs that spawn (Pot Odds + Outs + Equity + EV for Caller, Fold Probability + Bet Size + EV for Aggressor) gives the user two redundant cues for which type of decision they are evaluating, neither of which requires reading explicit text labels. This design intentionally trains the user's eye to read poker state from visual cues, which is the same skill they will need to apply at a real table.

### Additional HUD Elements

The pot size, blinds, and position indicators are displayed at appropriate positions around the table edges. Specific placement is determined during the visual implementation pass. The Ghost Timer (Module 6) is also displayed as a HUD element, with placement determined during the visual implementation pass.

### Leave-Drill Confirmation Flow

Clicking the X icon opens a confirmation popup overlay. This popup asks the user whether they want to leave the current scenario. The popup contains two action buttons: a Yes button (red color) confirming the user wants to leave (and returning them to the Mode Selection screen), and a No button (grey color, default keyboard focus) dismissing the popup and continuing the current game. Pressing the Escape key on the keyboard is treated as if the user had clicked No. Clicking Yes returns the user to the Mode Selection screen, NOT to the Root screen. This is intentional. The X is designed for users who want to switch between drill modes mid-session, which is the most common reason to leave a scenario in progress. To return all the way to the Root screen, the user clicks the Home icon (which is again present in the top-right cluster) from the Mode Selection screen.

### Persistent Top-Right Cluster Behavior

The Settings, Shop, and Help icons remain in the top-right cluster across the Mode Selection screen, the Game screen, and the Post-Round Screen. Their behavior is identical across all screens: clicking any of the three opens a centered modal popup overlay, not a new screen. The fourth icon in the cluster varies by screen. On the Mode Selection screen and on the Post-Round Screen, it is the Home icon. On the Game screen (during an active scenario), it is the X icon.

### Modal Popup Conventions

All modal popups in the application follow a consistent layout: a centered overlay on the screen, an X close button in the top-right of the modal itself (this is the modal's own close button, distinct from the persistent X icon in the top-right cluster), and dismissibility via clicking outside the modal area. The persistent cluster modals (Settings, Help, Shop) include an identifying header in the top-left of the modal: the corresponding icon (cog, ?, or storefront) followed by the modal's name. The name renders inside a rounded pill-shaped background tint using bg_button_default as the fill color, with the name text in text_primary inside the pill. The icon sits to the left of the pill, not inside it. Confirmation modals (leave-drill, leave-site, section reset) and the Custom Mode Configuration popup do not include the icon-plus-name header — their body content provides sufficient context. When a modal opens over the Game screen during an active scenario, scenario timing pauses for the duration the modal is open (see Module 6's Delta Timer). During an active tutorial, modals open in a locked state — interactive controls render at approximately 40% opacity and clicks are silently suppressed, while scrolling and the modal's X close button remain functional. A small banner at the top of the modal reads "Locked during tutorial — close to continue." See Notes — Tutorial System. All confirmation modals (leave-drill, leave-site, delete account, skip tutorial, reset all settings, reset section confirmation) share a uniform focus list, registered with focus_manager on modal open via push_focus_context: No → Yes → X close, wrapping from X close back to No. Default focus pointer on modal open is 0 (No, the safe default). Tab advances through this list; shift-tab reverses. Enter on the focused button activates it (No dismisses the modal, Yes confirms the destructive action, X close dismisses without action). Escape dismisses the modal per the existing Notes — Escape Key Behavior spec.

### Game Screen Aesthetic

The game screen establishes a stylized high-limit poker room atmosphere. Explicitly NOT a Las Vegas casino aesthetic — no slot machines, no neon, no overt gambling-floor visual language. The space should evoke a private, premium card room: dim lighting, leather and wood textures, brass fixtures, and a sense of restraint.

### Dealer Asset

A stylized character occupies the dealer position. The dealer is rendered in a Persona-character visual idiom — illustrated, stylized, not photorealistic human. However, the dealer character's stylization does not extend to the broader UI; the application does not use the explicit Persona-game UI aesthetic (the diagonal slashes, the bold red/yellow color blocks, the explicit anime-style transitions) anywhere else. The dealer wears butler-formal attire, has a sharp and defined mustache, and wears a golden monocle. He is faceless except for a sharp mustache, a golden monocle, and visible eyebrows — no eyes, nose, or mouth are rendered.

### Poker Table

The poker table is the centerpiece of the game screen. Its render quality must match the seriousness of the application — integrated visually with the room background rather than presented as a flat sticker overlay. The table sits in the room with appropriate lighting and depth, not as a 2D image pasted on top of the background.

### Background Treatment

The room behind the dealer and table is rendered with depth-of-field treatment — blurred or faded (the specific technique is determined during the visual implementation pass). The intent is to keep visual focus on the table and dealer while preserving an atmospheric sense of place.

### Background Continuity Across Screens

The high-limit room background renders on all four screens (Root, Mode Selection, Game, Post-Round) using three pre-rendered blur variants of the same source image. This creates visual continuity and a sense of deepening presence in the room as the user navigates inward, with no real-time blur computation cost. Root screen: uses background_root.png (heavily blurred, ~12-20 pixel radius equivalent). Background is purely atmospheric. Buttons dominate the visual hierarchy. Mode Selection screen: uses background_mode.png (medium blur, ~6-10 pixel radius equivalent). Background becomes legible. Room shapes and silhouette movement become visible. The STANDARD button and the three middle buttons remain the focal point. Game screen: uses background_game.png (light blur, ~2-4 pixel radius equivalent). The high-limit room is recognizable. The poker table and dealer render sharp on top of this background; only the room itself retains blur. Screen transitions crossfade between blur variants over ~300ms ease-out, synchronized with the existing button morph animation. The crossfade visually approximates a camera-focus-pull effect without requiring runtime blur computation, ensuring consistent performance across all devices including low-end mobile browsers. Asset note: All three backgrounds derive from a single source render of the high-limit room, with blur applied during asset preparation rather than at runtime. Estimated bundle weight: ~1.3 MB combined for all three variants.

### Background Movement (Atmospheric Layer)

The blurred background may include occasional silhouette movement suggesting other patrons in the high-limit room. Implementation uses two to three silhouette sprites drifting at slow speeds (one to two pixels per second) with heavy blur and low alpha. The intent is ambient suggestion of presence — never literal animated characters or visible figures. Movement should register only on extended observation, not on first glance. A user glancing at the screen should not consciously see anyone moving; a user staring at the screen for a sustained period should sense subtle activity in their peripheral vision.

## Notes — Canvas Layout and Scaling

The application renders entirely to a single HTML5 canvas element. This section defines how the canvas dimensions relate to the browser window, how elements within the canvas are positioned across different window sizes, and how the layout handles edge cases (extreme aspect ratios, very small windows, mobile devices).

### Canvas dimensions match window dimensions

The canvas element's pixel dimensions are continuously set to match the browser viewport's dimensions. The canvas always fills the available browser space; there are no black bars or letterboxing around the canvas. On window resize, the canvas dimensions update on the next frame to match the new viewport size. The layout re-renders at the new dimensions in the same frame; there is no transitional state, no animated resize, no flicker.

### Relative element positioning

All visual elements are positioned relative to canvas edges and regions, not to absolute pixel coordinates. Dear ImGui's relative-layout primitives (alignment, padding, content sizing, group bounding, anchor-to-edge positioning) handle the per-frame layout computation based on the current canvas size. Spatial language used throughout the spec (top-left region, middle, top-right cluster, bottom-right, etc.) refers to canvas regions, not pixels. Each region's boundaries are computed at render time from the current canvas dimensions. Elements within a region are positioned with padding and spacing values that scale proportionally with the canvas size. Text and icon sizes scale uniformly with the canvas. When the canvas is larger (e.g., on a 4K monitor), text, buttons, and icons render at a larger pixel size while maintaining the same proportional relationship to the rest of the layout. When the canvas is smaller, everything scales down proportionally. This is uniform scale, not logical-size preservation — the visual experience zooms in or out as a whole rather than reflowing into a different layout.

### Aspect ratio handling

The application does not enforce a specific canvas aspect ratio. Elements reposition automatically to use the available space at whatever aspect ratio the browser window presents. At standard 16:9 windows (most laptops, monitors, common TV sizes), the layout renders as designed with elements at their nominal positions and spacing. At ultrawide aspect ratios (21:9, 32:9), the canvas extends horizontally. The center stage of the screen (poker table, dealer, math input zone) remains centered. The persistent cluster anchors to the top-right edge, the chip legend to the top-left edge, with additional horizontal space distributed as padding. The layout remains functional and visually balanced. At taller aspect ratios (4:3, 5:4) or narrower windows, the canvas extends vertically or compresses horizontally. Elements maintain their region anchors. The layout may feel slightly more vertical than designed but remains usable. At extreme aspect ratios (square 1:1, very narrow vertical strips), the layout may compress or stretch awkwardly but continues to function as long as the canvas size exceeds the minimum size threshold below.

### Minimum size threshold

Below a minimum canvas height of approximately 600 pixels, the application's visual hierarchy degrades to unreadable. At this threshold, the canvas does not attempt to render the standard layout. Instead, a centered message displays: "Please use a larger window." The message uses bg_primary as the surrounding fill, text_primary for the message text, and no other elements render. The threshold is checked on every resize event. When the user enlarges their window past the threshold, the standard layout resumes rendering immediately on the next frame.

### Mobile devices

Mobile portrait and mobile landscape orientations fall below the minimum size threshold on most devices, and additionally have aspect ratios significantly different from desktop. Mobile devices receive a dedicated fallback screen instead of attempting to render the standard layout. Detection: a User-Agent check at app launch identifies mobile browsers (iOS Safari, Android Chrome, etc.). On detection, the canvas renders a centered message: "This trainer is designed for desktop. Please visit on a laptop or desktop browser." The message uses the same styling as the minimum-size message. No mobile-specific layout is provided. Mobile support is a known future enhancement, not part of V1.

### Asset resolution

All raster assets (dealer character PNGs, table, chip denomination textures, card faces, UI icons, background blur variants) are exported at a resolution sufficient to render crisply at canvas dimensions up to 4K (3840×2160). Assets that would scale up significantly on 4K displays without high-resolution source files would render with visible softness or pixelation. Asset export targets at minimum 2x the nominal canvas size used during design (i.e., assets exported at 3840-pixel-equivalent width if the nominal design canvas is 1920px wide). Bitmap fonts in Dear ImGui are re-rasterized at the requested size each frame; font scaling is handled automatically by the rendering layer with no asset preparation required.

### Architectural commitment

No element in the layout uses absolute pixel coordinates. All positions, sizes, and spacing values are expressed as functions of the current canvas dimensions or as proportions/anchors via Dear ImGui's layout primitives. The single canvas-dimensions-to-window-dimensions binding is the source of all sizing in the application; everything else derives from it.

## Notes — Color Tint Theme

The application uses a token-based color theme system in which every UI element references named color tokens rather than hardcoded color values. Switching the active theme swaps the underlying token values, and the entire UI re-renders in the new palette without any per-element changes. This is the standard CSS-custom-properties pattern adapted to the application's Dear ImGui rendering layer.

### Core Architectural Constraint

No element in the UI may hardcode color values. Every color-bearing element — buttons, button backgrounds, button text, button hover states, modal backgrounds, modal borders, input box outlines, input box focus outlines, input box text, table felt tint, time grade text colors (overtime/undertime), pass/fail color coding on the stat modal, separator lines, progress arc on the loading screen, and all other rendered colors — must reference one of the named tokens defined below. Implementation enforces this through the rendering layer; if a color value appears anywhere in code other than the theme definition, it is a violation of this architecture.

### Implementation Layer (Dear ImGui)

Dear ImGui exposes a global style structure containing color slots accessed via ImGui::GetStyle().Colors[ImGuiCol_X]. The application defines a Theme struct in C++ that maps the token names below to ImGui color slots. At application startup and on every theme change, the active Theme struct's values are written into the ImGui style. All ImGui::PushStyleColor, ImGui::Button, ImGui::InputText, etc. calls read from these slots. Theme switches are O(1) — a single struct copy followed by a single ImGui style refresh. For elements rendered outside Dear ImGui's standard widgets (custom-drawn elements like the dealer button progress arc, the side-pot stacked-chip icon, the stat modal three-column layout, table felt tint, custom button graphics), the rendering code reads directly from the active Theme struct in C++ rather than from ImGui's style array. The Theme struct is the single source of truth; ImGui's style array is one of two consumers of it.

## Notes — Communication Backbone

The application uses a centralized communication backbone that handles cross-zone coordination concerns: input routing, lifecycle events, screen state, and frame-level timing. This backbone exists because several behaviors in the application are inherently global (escape key routing, scenario lifecycle events, modal pause signals, animation tick) and would otherwise require zones to know about each other's internals to coordinate. The backbone makes every zone independent: each zone registers its handlers and subscribes to relevant events without needing to query or call into other zones directly. The backbone consists of six primitives, all defined in shared headers loaded by every zone:

### Global Event Router

A centralized keyboard and mouse event dispatcher. The main loop receives raw browser events and forwards them to the router, which holds the current input context (active screen, open modals, tutorial state) and dispatches events to registered handlers via a priority stack: Tutorial overlay handlers (highest priority — captures escape, enter, etc. when tutorial is active) Topmost modal handlers (modals always capture before underlying screens) Active screen handlers (root, mode selection, game, post-round) Global default handlers (no-op fallbacks) Zones register handlers via register_handler(context, key, handler_fn) where context is a state predicate (e.g., "current screen is Game AND no modal is open AND tutorial is not active"). The router evaluates the priority stack on every keypress, finds the topmost matching handler, and dispatches. This pattern eliminates the need for zones to know about each other. Zone 11 (modals) doesn't need to know that Zone 08 (Game Screen) handles escape differently — the router handles the layering.

### Scenario Lifecycle Event Bus

A typed event channel for scenario-level events. Zone 01 (Engine) fires named events at well-defined points in the scenario lifecycle: scenario_spawned(scenario_state) — fires after a new scenario is generated, before rendering begins scenario_submitted(grading_result) — fires when the user presses Enter to submit answers scenario_ended() — fires when the user navigates away from a scenario (via Again commit, Exit, Home, or leave-drill) tier_advanced(tier_n) — fires within a multi-tier Aggressor scenario when the user moves between tiers Any zone can subscribe to these events via on_scenario_spawned([](const ScenarioState& s){ ... }). Subscriptions are global; events fan out to all subscribers. Consumers across the application: Zone 03 (Audio) subscribes to scenario_spawned to play the Card Deal SFX and orchestrate audio choreography Zone 10 (Temporal) subscribes to scenario_spawned to start the Delta Timer and scenario_ended to stop it Zone 09 (Math Interrogator) subscribes to scenario_spawned to reset its input state and clear previous answers Zone 08 (Game Screen) subscribes to scenario_spawned to render the new table state Zone 13 (Post-Round) subscribes to scenario_submitted to receive the grading result and trigger the slide transition The bus is the single source of truth for "what is the engine doing right now." Zones never call into Zone 01 to ask; they listen for events.

### Screen State Singleton

A globally-readable record of which screen is currently active. Tracked values: Current screen: Root | ModeSelection | Game | PostRound | TutorialComplete Active scenario: optional (None if not on Game or Post-Round) Tutorial state: Inactive | Prompt | Active(step_n) | Complete Updated by transition triggers (button morph completion, ceremonial transition completion, slide transition completion) and read by anyone who needs to make a state-dependent decision (the event router uses this; zones with conditional behavior use this). Read-only for everyone except the transition orchestrator.

### Modal State Observer

A query interface and event channel for modal lifecycle: Queries: is_any_modal_open() → bool, current_modal_id() → optional, is_modal_locked() → bool (true during tutorial) Events: modal_opened(modal_id), modal_closed(modal_id) Owned by Zone 11 (Modal Infrastructure), but the query and event interface is part of the communication backbone so other zones can react. Specifically, Zone 10 (Temporal) subscribes to modal_opened/modal_closed to pause and resume the Delta Timer per the Modal Popup Conventions.

### Global Animation Clock

A shared monotonic time source for all frame-level animations. Exposes: total_ms_since_app_start() → uint64_t — for absolute timing (e.g., "the dealer arrived 600ms ago") delta_ms_since_last_frame() → float — for frame-rate-independent animation steps Owned by Zone 05 (Bridge / Main Loop) which advances the clock once per frame. Every zone with animations reads from this clock rather than maintaining its own time tracking. This guarantees that all animations across all zones complete at the same wall-clock duration regardless of frame rate, satisfying the Reduce Motion accessibility requirement and the consistent-pacing principle in Module 3's Main Loop spec.

### Focus Manager

A centralized keyboard focus management system. Because the application renders entirely to a single HTML5 canvas element, the browser cannot provide its default tab-navigation, focus-state, or keyboard activation behavior — the canvas appears to the browser as a single focusable element with no internal structure. The focus_manager replaces that browser-provided behavior with a custom equivalent. The focus_manager owns the following state: Active screen's focusable list — an ordered list of focusable element identifiers for the currently active screen. Each screen registers its focusable list during initialization or on screen transition. Modals that open replace the active list with the modal's own focusable list, with the underlying screen's list pushed onto a stack and restored on modal close (focus trap pattern). Focus pointer — an integer index into the active focusable list, indicating which element currently has focus. Default value on screen entry is 0 (first element in the list). Keyboard mode flag — a boolean tracking whether the user has activated keyboard navigation in the current session. Initially false. Set to true on the first tab keypress OR the first mouse click on any focusable element. Once true, persists for the remainder of the session and is never set back to false. Focus indicator visibility — the focus indicator only renders when the keyboard mode flag is true. Before keyboard mode is active, no focus indicator renders anywhere on screen, even if a focus pointer value exists internally. The focus_manager exposes the following operations: register_focus_list(screen_id, list) — called by each screen during initialization to register its ordered focusable element list. Modals call this on open with their own list, after pushing the underlying screen's list onto the focus stack. get_focused_element() — returns the focusable element id currently pointed to by the focus pointer. Used by the event router to determine which element receives enter-key activation. advance_focus(direction) — increments (tab) or decrements (shift-tab) the focus pointer with wrap-around at list boundaries. snap_focus_to(element_id) — sets the focus pointer to the given element. Called when the user mouse-clicks any focusable element so that subsequent keyboard navigation continues from that element's position. Also activates keyboard mode if not already active. activate_keyboard_mode() — sets the keyboard mode flag to true. Called automatically by the first tab keypress and the first mouse click on a focusable element. Idempotent — subsequent calls are no-ops. push_focus_context(new_list) / pop_focus_context() — used by the modal infrastructure to swap focus lists on modal open/close, preserving the underlying screen's focus state. The focus_manager interacts with the Global Event Router as follows: the event router dispatches tab and shift-tab keypresses to focus_manager.advance_focus, dispatches enter keypresses to the currently focused element's handler (resolved via get_focused_element), and the focus_manager calls activate_keyboard_mode on first tab/click. The event router does not directly manipulate focus state; all focus mutations flow through the focus_manager API. Mouse hover never affects the focus pointer. The focus pointer is decoupled from cursor position. Only explicit clicks or keyboard navigation move the pointer. The focus_manager is responsible for emitting a focus indicator render call each frame for whatever element the pointer currently points to, but only when keyboard mode is active. The actual rendering of the indicator is performed by the rendering layer, which queries the focus_manager for the current focused element id and its position.

### Header Locations

All six primitives are declared in shared headers under src/backbone/: src/backbone/event_router.hpp src/backbone/scenario_events.hpp src/backbone/screen_state.hpp src/backbone/modal_state.hpp (interface only — implementation lives in Zone 11) src/backbone/animation_clock.hpp (interface only — implementation lives in Zone 05) src/backbone/focus_manager.hpp These headers are loaded by every zone. They are the contract surface for cross-zone coordination.

### Initialization Order

The backbone primitives initialize in a fixed order during boot, before any zone-level initialization runs: Animation clock (no dependencies) Screen state singleton (no dependencies) Event router (depends on screen state for context evaluation) Focus manager (depends on event router for keyboard event routing) Scenario lifecycle event bus (no dependencies) Modal state observer (depends on event router for keyboard event capture and on focus manager for focus context push/pop) After backbone initialization completes, zone-level initialization begins (asset loading, persistence reconciliation, etc.). Zones register their handlers and subscriptions during their own init phase.

### Architectural Forbidden List

To preserve the backbone's invariant: No zone may directly call into another zone's internals to query state or trigger behavior. All cross-zone coordination flows through the backbone. No zone may maintain its own time tracking for frame-level animations. The animation clock is the single source of truth. No zone may register a global keyboard handler outside the event router. All keyboard handling is mediated. No zone may directly manipulate the focus pointer or focus indicator state. All focus mutations flow through the focus_manager API (snap_focus_to, advance_focus, push_focus_context, pop_focus_context). No zone may render its own focus indicator. The rendering layer queries focus_manager for the currently focused element id and renders the indicator from that single source of truth. No zone may fire scenario lifecycle events directly to consumers. Events flow through the bus, never zone-to-zone. The backbone primitives are read-only contracts from each zone's perspective. The backbone implementation lives in Zone 05 and Zone 11; the interfaces are Phase 0 and immutable post-Phase 0.

## Notes — Settings Page Specification

Order: Gameplay → Units → Display → Audio → Recap → Tomatoes → Account → General → Legal

### Settings Page

The settings page presents all settings in a single scrollable list, sectioned by category. A category sidebar allows the user to jump directly to any section. A search function performs case-insensitive matching against setting names with typo tolerance via a fuzzy-match library such as Fuse.js. All settings autosave on every change — there is no Save or Apply button required. Settings modal focus list, registered with focus_manager on modal open via push_focus_context, structured as a continuous traversal across three logical zones: search input → sidebar sections (top to bottom in the sidebar order: Gameplay → Units → Display → Audio → Recap → Tomatoes → Account → General → Legal) → body settings controls (top to bottom in the order they appear in the body, traversing all sections continuously from first to last) → X close, wrapping from X close back to the search input. Default focus pointer on modal open is 0 (search input). Tab advances through this list; shift-tab reverses through the same list in reverse order. Enter on a focused sidebar section scrolls the body to that section AND moves the focus pointer to the first setting control of that section in the body. Mouse click on a sidebar section has the same effect (and activates keyboard mode if not already active). This means a user can navigate to a section either by tabbing through the sidebar and pressing enter on the desired section, or by clicking the section directly with the mouse — both methods produce identical state changes. A user who tabs through every sidebar section without pressing enter on any of them will eventually reach the end of the sidebar zone and tab into the body zone, landing on the first setting control of the first section. From there, continued tabbing advances through every setting in every section in body order, eventually reaching the X close. From X close, the next tab wraps back to the search input. For all slider-based settings, after the user releases the slider and the value autosaves, a faded/ghost representation of the default value is rendered alongside the current value, so the user can see at a glance what the original default was and how far they have deviated from it. Search empty state: when the search query returns zero matches, the modal body renders blank — no setting controls, no placeholder text. The sidebar remains visible. Clicking any sidebar section while a search is active clears the search query, restores the full Settings list, and jumps to the clicked section.

## Notes — Post-Round Screen

The Post-Round Screen is the dedicated screen the user transitions to after submitting their answers via Enter on the Game screen. It is structurally a screen, not a modal — the Game screen is replaced rather than overlaid.

### Background

The Post-Round Screen uses background_mode.png (medium blur, ~6-10 pixel radius equivalent), the same blur variant used by the Mode Selection screen. This signals that the user has stepped out of the active Game state into a reflective space.

### Screen Regions and Layout

- **Top-left region:** Scenario ID display. A small grey text label reads "Scenario ID: [64-bit number]" (e.g., "Scenario ID: 4729183746281"). Beneath the text label, two utility buttons stack vertically, each scaled to match the width of the Scenario ID text line. The first button uses a traditional two-papers Copy glyph; the second uses a traditional square-with-up-arrow Share glyph. These buttons are smaller than the Again button, reflecting their secondary utility role.
- **Top-middle region:** empty, except that the dealer's face peeks above the top of the stat modal (see Dealer Asset section below).
- **Top-right region:** the persistent four-icon cluster (Shop, Help, Settings, Home). The fourth icon is Home (not X), since the user is no longer in an active scenario.
- **Middle region:** the stat modal (see Stat Modal section below).
- **Bottom-left region:** Exit button rendered as a door glyph. Clicking Exit returns the user to the Mode Selection screen via the standard ceremonial transition.
- **Bottom-right region:** Again button, larger than the Copy/Share/Exit buttons. The text label is "AGAIN" by default and changes to "CONFIRM" when the button enters its armed state. Post-Round focus list, registered with focus_manager on screen entry. The list composition varies by scenario type: For multi-tier Aggressor scenarios (tier tab strip present): tier tab strip (single tab stop) → Again → Exit → Copy → Share → Shop → Help → Settings → Home, wrapping from Home back to the tier tab strip. Default focus pointer on screen entry is 0 (tier tab strip). For non-multi-tier scenarios (Caller scenarios and single-tier Aggressor scenarios; no tier tab strip rendered): Again → Exit → Copy → Share → Shop → Help → Settings → Home, wrapping from Home back to Again. Default focus pointer on screen entry is 0 (Again). The tier tab strip is a single tab stop containing the per-tier tabs (Tier 1, Tier 2, Tier 3, Tier 4, Summary). When the strip has focus, the focus indicator renders as a 2px outline using border_focus around the bounding box of the entire tab strip. Left and right arrow keys navigate between tabs within the strip (Tier 1 → Tier 2 → ... → Summary, wraps at ends). Keys 1-5 select the corresponding tab directly (1 = Tier 1, 2 = Tier 2, 3 = Tier 3, 4 = Tier 4, 5 = Summary). Tab swaps focus to Again (out of the strip); shift-tab wraps backward in the focus list. Tab swap occurs on focus change within the strip, not on enter — enter on a focused tab strip is a no-op (the tab is already selected by being focused). Mouse click on any tab selects that tab and snaps the focus pointer to the strip.

### Dealer Asset

A front-facing variant of the dealer character occupies the center of the screen, behind the stat modal. The dealer is centered horizontally on the screen with approximately 10% of vertical screen space remaining above his head — his face does not touch the top of the browser viewport. The dealer is rendered at the same scale as the side-profile dealer used on the Game screen, with slight upward scaling permitted if necessary to maintain the 10%-from-top constraint. The front-facing dealer maintains all existing identity markers from the side-profile butler: golden monocle, sharp mustache, faceless otherwise. He has visible eyebrows.

### Performance Expression

The dealer's expression reflects the user's performance on the just-completed scenario, set when the scenario ends and held constant throughout the user's time on the Post-Round Screen (including across tier tab navigation): Pass condition (neutral expression): All math inputs satisfy the accuracy check AND total scenario time is at or under the user's target time. This is the same threshold that triggers tomato awards. Fail condition (raised-monocle expression): Anything other than the pass condition. The dealer's monocle-side eyebrow is raised; the monocle on that side may be slightly tilted. Optional subtle forehead wrinkle on the raised side. The butler dealer uses two complete asset variants for these states (butler_neutral.png and butler_raised.png). The asset swap occurs at scenario completion before the dealer fades in.

### Frog State

If the user has activated the Frog dealer easter egg, the dealer rendered on the Post-Round Screen is the Frog (always front-facing — the Frog has only one orientation in this application). The Frog's expression is rendered via overlay layers on top of the base Frog asset: Pass condition (blushing): Two small light pink circles render on the Frog's cheeks. Implemented as a transparent overlay PNG (frog_expression_pass.png) layered over the base Frog asset. Fail condition (tongue out): The Frog's tongue protrudes downward in a small mocking expression, sized proportional to the Frog. Implemented as a transparent overlay PNG (frog_expression_fail.png) layered over the base Frog asset. If the Frog expression overlays have not yet finished loading when the user reaches the Post-Round Screen, the base Frog renders without expression. When the overlay completes loading, it fades in over approximately 200ms on the next frame boundary.

### Stat Modal

The stat modal is a centered overlay on the screen, rendered at 65% opacity (35% transparent, reading as a slightly greyish translucent panel). The top edge of the modal sits immediately below the dealer's face with a small vertical gap. The modal extends downward from that point with vertically symmetric proportions relative to the screen. Modal dimensions are fixed; the modal is compact-by-default and scrollable only if the recap content overflows.

### Tabs (multi-tier Aggressor scenarios only)

For multi-tier Aggressor scenarios (when the Bet Sizing Engine setting is enabled), the top of the stat modal contains a horizontal row of tier tabs — one tab per tier (Tier 1, Tier 2, Tier 3, Tier 4) plus a "Summary" tab. The currently-viewed tab is visually highlighted. Clicking a tab swaps the modal body content to display that tier's recap. For non-multi-tier scenarios (Caller scenarios, single-tier Aggressor scenarios with the Bet Sizing Engine setting disabled), no tab strip is rendered. The modal displays the recap content directly. Default landing tab for multi-tier Aggressor scenarios is configurable via the Recap Settings section (default: Tier 1).

### Tier Tab Contents

When viewing a tier tab, the modal body uses a three-column layout (with no visible column dividers) and subtle 1-pixel-wide row separators at low opacity (10-15%) in a neutral color: Column 1 — Input name (e.g., "Pot Odds", "Outs", "Equity", "EV", "Fold Probability", "Bet Size") Column 2 — Correct answer with grading deviation (e.g., "$120 ± $6") Column 3 — User's submitted answer, color-coded green if within margin, red if outside margin Bet-size-independent inputs (Equity if Called, Outs) are echoed at every tier tab to preserve visual symmetry, even though the user only submitted them once during the scenario. Above the Overall row, an additional row displays the Time Grade for that tier in the format: "Target Time: 15s | Actual Time: 22s (+7s Overtime)". The Actual Time portion is rendered red if overtime, green if undertime. During tutorial scenarios, the Time Grade row renders the placeholder text "Tutorial — timer disabled" in text_secondary instead of the standard format, since the Delta Timer is disabled during tutorials (see Notes — Tutorial System). The bottom row of the modal is the Overall row. Column 1 reads "Overall". Column 2 is empty (there is no "correct answer" for an aggregate). Column 3 displays the user's overall accuracy as a percentage across all inputs in this tier. If the scenario was a side-pot scenario, a small icon renders next to the word "Overall" in Column 1. The icon depicts three chips stacked vertically with a fourth chip leaning diagonally against the stack. All four chips are rendered in distinct colors.

### Summary Tab Contents

The Summary tab provides a scenario-level overview rather than tier-level detail. Its layout differs from the tier tabs (no three-column structure): Overall scenario accuracy: total inputs correct out of total inputs across all tiers, displayed as a fraction and percentage (e.g., "9/12 correct (75%)"). Per-tier breakdown: a compact row showing accuracy at each tier (e.g., "Tier 1: 3/3 | Tier 2: 2/3 | Tier 3: 3/3 | Tier 4: 1/3"). Time Grade: target time vs. actual time across the full scenario, with overtime/undertime color coding consistent with the tier tabs. Side-pot indicator: if the scenario was a side-pot scenario, the same chip-stack icon renders alongside the scenario summary.

### Again Button Behavior

The Again button uses a double-confirmation interaction model to prevent accidental skips: First press (via Enter key OR mouse click): the button enters armed state. Visually, the button color darkens; the text label changes from "AGAIN" to "CONFIRM". The button remains armed until either committed or cleared. Second press (via Enter key OR mouse click while armed): the button commits. The Post-Round Screen transitions to a fresh new scenario (with a new Scenario ID, generated per Module 1's Scenario Generator) on the Game screen. The armed state persists indefinitely; there is no timeout. The user may idle on the Post-Round Screen for any duration without losing their place in the recap. Mouse click behavior is parity with keyboard: a single click arms the button, a second click commits. If the user clicks Exit while the Again button is in armed state, Exit operates independently and the armed state is irrelevant. The user transitions to Mode Selection.

### Scenario ID Utility Buttons

The Copy button copies the raw Scenario ID number (e.g., "4729183746281") to the system clipboard. Brief visual feedback indicates success (icon flash or "Copied" tooltip for approximately 1 second). The Share button uses the Web Share API on platforms that support it (primarily mobile), invoking the native system share sheet with the URL form of the scenario (e.g., "[your-domain]/?scenario=4729183746281"). On platforms without Web Share API support (most desktop browsers), the Share button falls back to copying the URL to the clipboard with the same visual feedback as Copy. Clipboard API and Web Share API failure handling: If the clipboard API (navigator.clipboard.writeText) is unavailable or throws an error when invoked by Copy, or if both the Web Share API and the clipboard fallback fail when invoked by Share, a unified fallback modal opens. The modal is a centered overlay with fixed dimensions, containing: a single line of text reading "Copy this:" using text_primary, a single-line text field below the label rendered using bg_input and border_default, pre-populated with the text the user attempted to copy or share (the raw Scenario ID for Copy failures, the scenario URL for Share failures), and an X close button in the top-right of the modal per standard Modal Popup Conventions. The text field is selectable and editable (user can select all and copy manually); the field auto-selects its contents on modal open so the user can immediately use the OS-native copy gesture (Cmd-C or Ctrl-C). The modal has no other buttons; the X close button is the only dismiss control. The fallback modal is keyboard-accessible per the standard modal focus context push.

### Transitions Onto and Off the Post-Round Screen

The Game screen transitions to the Post-Round Screen via a 350ms slide animation (right-to-left), accompanied by the Slide In SFX (see Module 2 audio specification). The Post-Round Screen transitions back to the Game screen (via the Again button commit) via a 350ms slide animation (left-to-right), accompanied by the Slide Out SFX. These two slide SFX are designed to complement each other tonally as a pair representing the recap-loop rhythm. The Post-Round Screen transitions to Mode Selection (via the Exit button or via the persistent cluster Home icon) via the standard 1.5-second fade-to-black-and-back ceremonial transition, identical to the Mode Selection → Game transition (see Transitions section below).

### Dealer Arrival Animation

When the Post-Round Screen renders, the screen content arrives in a sequenced fade-in over approximately 2 seconds total: Phase 1: The Game screen slides off via the 350ms slide animation. The Post-Round background (Mode Selection blur variant) is now visible. Phase 2: The front-facing dealer fades in over approximately 600ms. Phase 3: The stat modal fades in over approximately 600ms after the dealer has finished fading in. The dealer arrival animation is toggleable via the Recap Settings section (default ON). When the toggle is disabled, all elements (background, dealer, modal) appear simultaneously at the end of the slide transition with no sequenced fade-in. Audio on Post-Round entry beyond the Slide In SFX is silent. There is no chime, no correct/incorrect audio cue, no dealer voice line.

### Settings, Help, and Shop Modal Behavior on Post-Round

The persistent cluster icons (Shop, Help, Settings) on the Post-Round Screen behave identically to all other screens: clicking opens the respective modal as a centered overlay. The Settings modal renders and the Post-Round Screen content is preserved underneath. Clicking outside the modal or pressing the modal's X close button dismisses the modal and returns focus to the Post-Round Screen content. While any modal (Settings, Help, Shop) is open over the Post-Round Screen, keyboard input (including Enter) is captured by the topmost modal. Pressing Enter inside the open modal does not trigger the Again button.

## Notes — Tutorial System

The tutorial is an interactive overlay flow that walks first-time users through the application's UI and a complete pair of teaching scenarios (one Caller, one Aggressor). It is structurally an overlay layer that sits on top of the existing screens (Root, Mode Selection, Game, Post-Round) plus one terminal screen (the Tutorial Complete screen) that replaces the underlying app entirely. The tutorial does not replace any existing module's logic — it intercepts user clicks at specific points and routes the underlying engine through hardcoded scenario seeds, but the engine itself is unaware it is in a tutorial.

### Trigger and Detection

A boolean flag has_seen_tutorial_prompt persists in IDBFS, set to false on first session and set to true only when the user finishes the tutorial via the Tutorial Complete screen. The flag is checked specifically on Root screen render (not on app launch). If the user lands directly on the Game screen via a shared scenario URL (/?scenario=...), the tutorial prompt does not render — they play the shared scenario directly. The prompt only renders when the user reaches the Root screen for the first time in a session with the flag set to false. If false, the first-launch tutorial prompt renders; if true, no prompt renders. If the user re-launches the tutorial later via the Tutorial button at the bottom of the ? modal, the flag is set back to false at tutorial start and true again on completion. This means the flag's semantic is "the user has finished the tutorial at least once" — it gates the first-launch prompt only, not the manual re-trigger.

### First-Launch Prompt

A small modal renders centered on the Root screen on first session, after Tier 1 assets have loaded. The modal contains a brief introductory message ("New here? Take a quick tour to learn how the trainer works.") and two buttons: "Start Tutorial" (begins the tutorial overlay flow) and "Skip" (dismisses the prompt and sets has_seen_tutorial_prompt to true). Dismissal via clicking outside the modal or pressing Escape is treated as Skip. The modal follows standard Modal Popup Conventions but does not include the icon-plus-name header (the prompt is self-contained).

### Overlay Visual Pattern

When the tutorial is active, a grey lens overlay renders at approximately 60% opacity across the entire screen, with a cutout ("spotlight") around the currently-focused element that allows it to render at full visibility. A text callout renders adjacent to the spotlit element (positioned to avoid covering it), containing instructional text and a Next button. The user advances through tutorial steps via the Next button (mouse click) or Enter key. Pressing Enter outside of math-input contexts always advances to the next step. Within math-input contexts, Enter behavior is overridden as specified in the Math Input Walkthrough section below. A persistent "Skip Tutorial" text button renders at the bottom-center of the overlay layer (avoiding collision with the Again button at bottom-right of Post-Round and other screen content). The button is rendered using bg_button_default / bg_button_hover / bg_button_active with a text_button label. It is visible only while the overlay/spotlight is active — hidden during free-exploration moments (when the user is exploring an opened modal) and not present on the Tutorial Complete screen. Clicking Skip Tutorial opens a confirmation modal: "Skip tutorial? You can re-open it from the Help menu later." with buttons Yes (red, state_fail) and No (grey, bg_button_default, default focus). Confirming returns the user to the Root screen, sets has_seen_tutorial_prompt to true, and restores all forced settings (see Forced Settings section below). If the user closes the browser tab mid-tutorial, the flag remains false and the prompt re-appears next session. There is no resume — the tutorial restarts from the beginning.

### Modal Lock During Tutorial

During the tutorial, when the user is prompted to open a modal (Settings, Shop, Help, Custom), the modal opens normally but its interactive controls are disabled: All toggles, sliders, dropdowns, buttons, and input fields render at approximately 40% opacity. Clicks on disabled controls are suppressed silently — no shake, no flash, no tooltip. A small banner renders at the top of the modal with the text "Locked during tutorial — close to continue." Modals can be scrolled freely. Reading is unaffected. The modal's X close button and click-outside dismissal remain functional. Within the ? modal specifically, the Tutorial button at the bottom is also rendered at 40% opacity and is non-interactive during an active tutorial (clicking the Tutorial button while a tutorial is already running is a paradoxical state that the lock prevents).

### Hardcoded Scenario Seeds

The tutorial uses two pre-selected Scenario IDs that produce ideal teaching hands: TUTORIAL_SCENARIO_CALLER_SEED: a Caller scenario with clean numbers — round pot, simple outs count, approximately 33% pot odds. TUTORIAL_SCENARIO_AGGRESSOR_SEED: an Aggressor Pure Bluff scenario with clean numbers suitable for multi-tier bet sizing demonstration. These seeds are determined via a one-time scan of seed values during implementation and locked as C++ constants. The Scenario Generator (Module 1) does not need to know it is in a tutorial — when the tutorial intercepts a STANDARD click, it passes the appropriate seed to the engine, which generates the scenario deterministically as it would for any seed.

### Forced Settings

While the tutorial is active, the following settings are temporarily forced to known values, regardless of the user's actual settings: HUD: forced ON (so the user sees the floating bet amount and pot total). Countdown timer: forced OFF (no timer rendering during teaching). Delta Timer: disabled entirely during tutorial scenarios — no time tracking, no Time Grade computed, no overtime. The Time Grade row in the Post-Round stat modal during tutorial scenarios renders the placeholder text "Tutorial — timer disabled" in text_secondary. Reduce Motion: uses the user's existing setting value at tutorial start. Not modified during tutorial; user cannot change via Settings while tutorial is active. Theme: uses the user's existing setting value at tutorial start. Not modified during tutorial; user cannot change via Settings while tutorial is active. Units (Cash vs BB): uses the user's existing setting value at tutorial start. Not modified during tutorial; user cannot change via Settings while tutorial is active. When the tutorial completes (via Tutorial Complete screen or via Skip), all forced settings restore to the user's actual values from IDBFS. Dealer easter egg: disabled during tutorial. The dealer asset is rendered with an invisible click-blocking overlay during the tutorial that suppresses click registration on the dealer. Click counts do not accumulate. After tutorial completion, the overlay is removed and the easter egg behaves normally.

## Notes — Transitions

The application uses three distinct transition styles between screens, each chosen to match the semantic register of the navigation:

### Ceremonial Transitions (1.5-second fade-to-black-and-back)

Used for transitions that mark significant state changes: - Mode Selection → Game (committing to a session) - Post-Round → Mode Selection (via Exit or via Home icon, ending or stepping away from a session) - Game → Mode Selection (via leave-drill confirmation Yes button, leaving an active scenario) The transition fades the current screen to fully black over approximately 750ms, holds briefly, then fades into the destination screen over approximately 750ms. Total duration: ~1.5 seconds.

### Slide Transitions (350ms ease-out)

Used for the Game ↔ Post-Round loop, signaling lateral movement between paired states rather than ceremonial commitment: - Game → Post-Round: 350ms slide right-to-left, accompanied by Slide In SFX - Post-Round → Game (via Again commit): 350ms slide left-to-right, accompanied by Slide Out SFX Slide direction is consistent: rightward-flowing screen content moves off-screen to the left as new content slides in from the right (Game → Post-Round), and the symmetric reverse for Post-Round → Game.

### Crossfade Transitions (already specified, ~300ms)

Used for the existing button morph between Root and Mode Selection screens, including the synchronized crossfade between background blur variants. See "Background Continuity Across Screens" for the existing specification.

### Settings Toggle

A "Screen transitions" toggle in the Recap Settings section (default ON) controls whether ceremonial fade-to-black and slide transitions render their full animation or are replaced with instant cuts. When disabled, all screen-to-screen transitions occur instantly with no animation. The toggle does not affect modal open/close animations or the existing Root ↔ Mode Selection button morph (those remain controlled by the global Reduce Motion setting).

## Notes — Keyboard Focus Behavior

Because the application renders entirely to a single HTML5 canvas element, browser-default keyboard navigation (tab focus order, focus indicators, enter activation) does not function. The application implements its own equivalent via the focus_manager primitive (see Notes — Communication Backbone). This section codifies the global rules that apply across all screens; per-screen focus lists are documented in each screen's own section.

### Activation

Keyboard navigation is dormant until the user activates it. Activation occurs on either of two events: the first tab keypress in the session, or the first mouse click on any focusable element. Both events set the keyboard mode flag to true via focus_manager.activate_keyboard_mode. Once activated, keyboard mode persists for the remainder of the session and is never deactivated. Before activation, the focus indicator does not render anywhere on screen. Tab keypresses on the Root screen before any activation event are no-ops. Enter keypresses before activation are no-ops. After activation, the focus indicator renders on the element pointed to by the focus pointer, on every frame, until the user leaves the screen or a modal opens (in which case the modal's focus context replaces the screen's).

### Tab navigation

Tab advances the focus pointer to the next element in the active focusable list. Shift-tab decrements to the previous element. Both wrap at list boundaries — tab past the last element returns to the first, shift-tab past the first element jumps to the last. The active focusable list is the list registered by the topmost focus context — either the current screen's list, or, if a modal is open, the modal's list (with the screen's list preserved on the focus stack for restoration on modal close). The order of elements within each focusable list is documented in the per-screen and per-modal specifications. Where focus lists include a "group" (e.g., the bet size button row on the game screen, the tier tab strip on the post-round screen), the group occupies a single position in the list as one tab stop; navigation within the group is handled by other keys (number keys, arrow keys) as specified per group.

### Mouse interaction

Mouse hover does not affect the focus pointer. Hovered elements may render a separate visual hover state per existing spec (bg_button_hover, etc.), but this is unrelated to focus. Mouse click on any focusable element calls focus_manager.snap_focus_to with the clicked element's id. The pointer moves to that element regardless of its prior position. If keyboard mode was not yet active, the click also activates it. The click still performs its standard action (button activation, input field focus for text editing, etc.). The focus pointer update is a side effect of the click, not a replacement for it.

### Enter key

When keyboard mode is active, the enter key activates whatever element the focus pointer points to. The activation is equivalent to a mouse click on that element. The event router consults focus_manager.get_focused_element to identify the target. Specific elements override enter key behavior: Math input zone on the Game screen — enter submits all answers simultaneously across all visible inputs, regardless of which input is focused. This is the existing Module 5 spec and is unchanged. The enter-submits-all behavior overrides the standard "enter activates focused element" rule. Tutorial overlay during math input walkthrough — enter is suppressed until all visible inputs across all tiers are filled. Once all inputs contain values, enter submits per the standard behavior. Tab is the only advance mechanism between inputs during tutorial. This is the existing Notes — Tutorial System spec and is unchanged. Outside these contexts, enter behaves uniformly: it activates the focused element.

### Focus indicator visual

The focus indicator renders as a 2px outline using the border_focus token, drawn around the bounding box of the focused element. No fill change, no glow, no scale change — only the outline. The outline sits flush against the element's edge with no additional padding. For grouped focus elements (bet size button row, tier tab strip, range slider with two handles), the focus indicator renders around the bounding box of the group as a whole when the group has focus, not around individual buttons or handles within the group. The exception is the range slider, where each handle is a separate tab stop and the focus indicator renders around the focused handle individually. The user-facing color of the focus indicator is the border_focus token, which is theme-controlled. Each theme defines its own border_focus value (typically the same as accent_primary or a closely related color, ensuring visual consistency with the theme's accent palette).

### Focus context stack (modal focus traps)

When a modal opens, the modal infrastructure calls focus_manager.push_focus_context with the modal's own focusable list. The underlying screen's list and focus pointer are preserved on the stack. While the modal is open, tab and shift-tab navigate only within the modal's list — the underlying screen is unreachable by keyboard. This is the modal focus trap pattern. When the modal closes, focus_manager.pop_focus_context restores the underlying screen's list and focus pointer to their pre-modal state. The user resumes keyboard navigation from where they were before the modal opened. If multiple modals stack (e.g., a confirmation modal opens over the Settings modal), each push adds to the stack and each close pops in reverse order.

### Dealer not included in focus lists

The dealer asset on the Game screen and the Post-Round Screen is not included in any focusable list. The dealer is mouse-only for the easter egg interaction (22 consecutive clicks to toggle Butler/Frog). The 22-click counter is incremented by mouse click only; keyboard cannot trigger it. ​​

## Notes — Service Outage Banner

A shared banner system displays transient error messages when external services (Auth0, leaderboard server, etc.) are unreachable. The banner is the unified visual treatment for service-down events across the application; individual zones do not implement their own outage UI.

### Visual specification

The banner renders at the top-center of the canvas, approximately 1/6 of the canvas width. Height is approximately 60 pixels (scales with canvas size per the Canvas Layout and Scaling rules). The banner uses bg_modal for its background, text_primary for the message text, and a small icon to the left of the text indicating the nature of the issue (typically a warning glyph; the specific icon is determined during the visual implementation pass). The banner slides into view from above the canvas top edge over approximately 300ms with an ease-out curve. It remains visible for 5 seconds, then slides back up over approximately 300ms with an ease-in curve. Along the bottom horizontal edge of the banner, a white progress bar renders. The bar is the full width of the banner and approximately 2-3 pixels tall. At banner appearance, the bar is at full width. Over the 5-second visible duration, the bar's width animates linearly from right to left, reaching zero width as the banner begins its slide-out. The bar serves as a visual countdown indicating when the banner will dismiss automatically.

### Dismiss interaction

Clicking anywhere on the banner dismisses it immediately. On click, the banner begins its slide-out animation, the countdown bar animation halts, and the banner is removed from the screen at the end of the slide-out. Clicking does not navigate or activate any underlying element; the banner is a click-target only for its own dismiss action.

### Stacking behavior

The banner does not stack. If a second outage event fires while a banner is already visible, the existing banner is replaced in place by the new banner. The visible 5-second timer restarts; the countdown bar resets to full width. The slide-in animation does not replay (the new banner is already in the visible position from the previous banner).

### Triggers

The banner is triggered by service-down events, including but not limited to: Auth0 SDK health check fails immediately before opening the Sign In or Sign Up modal. The health check is performed synchronously when the user clicks Sign In, Sign Up, or any other button that would open an Auth0-dependent modal. If the health check fails (timeout, SDK error, network failure), the modal does not open; the banner displays instead. Message text: "Sign in temporarily unavailable. Please try again later." or "Sign up temporarily unavailable. Please try again later." depending on the source button. Auth0 health check fails on the Tutorial Complete screen when the user clicks Sign Up. Same banner behavior; the modal does not open. The user remains on the Tutorial Complete screen and can click Continue Playing or Home. Server-side leaderboard fetch fails: the leaderboard modal renders its own inline error message with a retry button (per Module 7's Leaderboard View spec). The banner is not used for leaderboard failures — those are handled inline in the leaderboard view. Future transient service errors as needed.

### Distinction from credential failures and other errors

Credential failures during Sign In or Sign Up (wrong password, invalid email, denied username, etc.) are not service-down events. These render inline in the relevant modal in state_fail styling per the existing modal specifications. The banner is reserved for service-unreachable events where the user cannot proceed regardless of input. Similarly, validation errors (empty required fields, password too short, etc.) render inline in the modal per existing specifications. The banner is not used for client-side validation feedback.

### Architectural ownership

The banner is owned by Module 4 (Dear ImGui Frontend) and rendered as a top-level overlay above all screens and modals. It is visible on every screen except the Tutorial Complete screen's "Continue Playing" or "Home" transitions in progress (i.e., during the ceremonial transition itself, the banner is suppressed; it resumes after the transition completes).

## Module 1: The Core Logic Engine (Pure C++)

The Global State Manager: Reads the user's top-level mode preference (STANDARD 50/50, AGGRESSOR 100%, CALLER 100%) and dictates the Scenario Generator's branching path. The Randomizer: Generates the scenario (Blinds, Street, Position) using the seeded RNG from the Scenario ID for deterministic reconstruction. Features a State Branching Matrix: Caller Branch: Opponent acts first and bets. Aggressor Branch: Action is on you to initiate. Subdivides into Pure Bluff, Value Bet, and Semi-Bluff. The Bet Sizing Engine: For Aggressor scenarios, it generates a multi-tier drill presenting the same scenario at various bet sizes (e.g., 1/3, 1/2, full, overbet). Toggleable via Settings (default ON). Deterministic Fold Function: The probability the opponent folds is computed deterministically to allow for learnable trade-offs: $P(\\text{fold})=\\text{clamp}(F+0.15\\times(\\text{bet_size_as_pot_fraction}-0.5),0.05,0.95)$, where $F$ is a randomized baseline tendency sampled from the seeded RNG within the user's current settings range at play time (default range $[0.2,0.8]$), NOT stored as part of the scenario. The F value is not displayed to the user on the Game screen. The user estimates fold probability from their configured difficulty range setting and from learned calibration over many scenarios — this mirrors real poker, where fold tendency is read from observed behavior rather than a displayed number. The Stack Size Generator: Assigns effective stack sizes to introduce Implied Odds calculations. The Side Pot Engine: Base percentage chance to trigger an "All-In" side pot split. The Deck Manager: Initializes a standard 52-card deck. Shuffles per scenario using the Mersenne Twister RNG seeded by the Scenario ID for reproducibility. Deals 2 hole cards to the user and community cards based on the spawned street (0 pre-flop, 3 on flop, 4 on turn, 5 on river). Maintains a remaining-deck state used by the Math Interrogator's Outs verification.

### Settings Application Pipeline

Settings are applied at well-defined points in the scenario lifecycle: At scenario spawn: All current settings are read from IDBFS and applied to the new scenario. This includes pre-evaluation settings (difficulty range, bet sizing engine toggle) which determine the math truth values for grading, and presentation settings (chip denomination mode, HUD visibility, theme, units) which determine rendering. Mid-scenario settings changes: If the user opens Settings mid-scenario and changes a value, the change applies on the next scenario spawn (next Again-button commit, or next session start), not retroactively to the current scenario. This applies uniformly to all settings. On replay (shared Scenario ID or saved scenario): The same locked properties reconstruct from the seed. All variable properties are re-evaluated against current settings, identical to a fresh spawn — meaning a replay can feel meaningfully different from the original play if settings have changed. Three categories of variable settings: Pre-evaluation settings affect math truth values: difficulty range (F value), bet sizing engine toggle. The True Evaluator uses these to compute correct answers and grading targets. Presentation settings affect rendering only: chip denomination mode, HUD visibility, countdown timer, theme, units, motion. These never change what is true, only what is shown. Screen-level settings affect transitions and recap behavior: Recap Settings, General Settings. These fire on screen transitions, not during gameplay. During an active tutorial, a subset of settings is temporarily forced to known values (HUD ON, Countdown timer OFF, Bet Sizing Engine ON, Delta Timer disabled) and restored to user values on tutorial completion. See Notes — Tutorial System for the full forced-settings list.

### Scenario ID & Reproducibility System

Every scenario generated by the engine is identified by a 64-bit unsigned integer Scenario ID. The Scenario ID is used as the seed for all randomness that defines the fundamental identity of the scenario. Given the same Scenario ID, the engine deterministically reconstructs the same scenario every time, in terms of: Hole cards (the user's two cards) Community board (the dealt portion of the five community cards based on the scenario's spawned street) Position (UTG, HJ, CO, BTN, SB, or BB) Scenario type (Caller, Aggressor Pure Bluff, Aggressor Value Bet, or Aggressor Semi-Bluff) Side-pot status (whether this is a side-pot scenario) Stack sizes (effective stacks for implied odds calculations) This separation enables three features at near-zero storage cost: Reproducibility for debugging: A specific scenario can be regenerated by its ID for inspection, bug reports, or test cases. Replay under different conditions: Past scenarios can be revisited and challenged under different current settings. A user can play a scenario originally under default conditions, then replay it later with tighter time pressure or hidden HUD elements to increase difficulty. The hand itself remains identical (same cards, same position, same scenario type, same side-pot status), but the user's experience of that hand changes based on current settings. This is a deliberate training feature: the same hand can be drilled at progressively harder conditions over time. Shareable scenarios across users: Two users with different settings can play the same scenario via a shared Scenario ID (e.g., "/?scenario=4729183746281" URL parameter). The cards are identical, but each user experiences the hand under their own training conditions. Friend with relaxed time pressure can share a scenario with another user playing under tight time pressure — same hand, different challenge. Storage cost: Each saved scenario is exactly 8 bytes for the Scenario ID, plus minor metadata such as timestamp and the user's accuracy/time score on that scenario. Tens of thousands of saved scenarios fit comfortably in browser local storage via the IDBFS persistence layer (Module 7). Implementation: Use std::mt19937_64 (Mersenne Twister) seeded with the Scenario ID. All randomness in scenario generation pulls from this seeded RNG so that the entire fundamental scenario reconstructs deterministically from the seed alone. The randomization algorithm is locked. It may not be changed without a scenario format version bump, to preserve replay compatibility for saved scenarios. Locked properties (encoded in seed): hole cards, community board, position, scenario type/branch, side-pot status, stack sizes. Variable properties (re-evaluated against current settings at play time): time pressure, HUD visibility, countdown timer visibility, multi-tier bet sizing display, chip denomination mode (Stake-scaled vs Fixed), opponent baseline tendency F value (re-rolled within current difficulty range setting, while remaining consistent with the locked scenario branch), audio settings, motion settings, color theme, units (Cash vs BB), and Recap Settings (dealer arrival animation, screen transitions, default tier tab). Any setting in the Display, Audio, Recap, Units, Tomatoes, or General categories is variable by definition — only Gameplay settings that affect scenario generation (street split weights) are seed-encoded. The True Evaluator: Calculates flawless math using strict Pot Conventions. Variable conventions (apply to all formulas): pot: chips in the middle BEFORE the current action's bets bet: size of the current single action (not cumulative across rounds) P(fold): probability opponent folds, computed via the deterministic fold function P(call): $1 - P(\\text{fold})$ equity: probability of winning at showdown given opponent calls (semi-bluff only) Formulas (LOCKED — V8.1 verified, do not modify):

```
Pure Bluff EV: EV = P(fold) × pot − P(call) × bet
Value Bet EV:  EV = P(call) × bet
Semi-Bluff EV: EV = P(fold) × pot + P(call) × [equity × (pot + 2 × bet) − bet]

Verified at extreme values:
  Pure Bluff at P(fold)=1.0:                                 EV = pot
  Pure Bluff at P(fold)=0:                                   EV = −bet
  Value Bet  at P(call)=1.0:                                 EV = bet
  Value Bet  at P(call)=0:                                   EV = 0
  Semi-Bluff at P(fold)=1.0:                                 EV = pot
  Semi-Bluff at P(call)=1.0, equity=0.4, pot=100, bet=50:    EV = $30
```

## Module 2: The Asset & Audio Pipeline (The Media Layer)

The Texture Loader (stb_image.h): Loads all PNG assets needed for game rendering: the background room in three pre-rendered blur variants (background_root.png for the Root screen, background_mode.png for the Mode Selection and Post-Round screens, background_game.png for the Game screen), the dealer character assets, the poker table integrated with the room, 52 individual card PNGs for hole and community cards, chip denomination textures for stack visualization, the dealer button graphic (a circular disc rendered in ocean blue, with a chip-style edge design featuring white rectangular inserts spaced around the perimeter, each containing a dice-face pattern; the center displays a stylized sage green M monogram representing the in-fiction venue's mark, used in lieu of traditional 'DEALER' text; an inset dashed ring sits between the dice-pattern edge and the central M, decorative on the table-rendered version and functional on the loading screen — see Module 3; ocean blue and sage green are intentionally not used in the play-chip color palette so the dealer button remains visually distinct from chips the user is wagering), all-in marker textures for side pot scenarios, UI iconography for the persistent top-right cluster (Shop, Help, Settings, Home/X icons), the Exit door glyph and the Copy and Share glyphs used on the Post-Round Screen, the side-pot stacked-chip icon used in the Post-Round stat modal Overall row when applicable, and the tomato icon used as the visual representation of in-app currency on the three currency-display surfaces (Shop modal, user profile/account view, and global leaderboard view). Chip denomination assets are eight PNGs total: $1, $5, $25, $100, $500, $1000, $5000, $25000. Chips are identified primarily by color and secondarily by edge pattern, consistent with real-cardroom convention. Chip faces do not display dollar values prominently — at most, a small denomination marking on the inlay, sized so it is not easily readable from typical viewing distance. The user identifies denominations by color, not by reading numerals off the chip. The denomination legend in the Game screen's top-left is the canonical color-to-value reference. Color assignment within any active denomination set follows standard US cardroom convention: smallest denomination is white, then red, green, black, purple, yellow/orange, brown/grey, gold/blue in ascending order. In Stake-scaled mode, the color-to-value mapping shifts per tier (e.g., white = $1 at Micro tier, white = $5 at Mid tier, white = $25 at High tier). In Fixed mode, the mapping is locked at white=$1, red=$5, green=$25, black=$100, purple=$500. The denomination legend in the top-left always reflects the active mapping. Exact hex values are determined during the visual implementation pass; the constraint is that all eight chip colors must be visually distinct from each other AND from the dealer button's ocean blue and sage green. Dealer Character Assets: The dealer character has two states (Butler and Frog) with distinct asset structures: Butler dealer assets (two complete PNG variants): butler_profile.png: side-profile view of the Butler, used on the Game screen during active scenarios. The Butler is rendered facing leftward (toward the table) in a dealing posture. butler_neutral.png: front-facing view of the Butler, used on the Post-Round Screen when the user has passed the scenario (all math inputs correct AND time at or under target). Both eyebrows are level, the monocle is straight. butler_raised.png: front-facing view of the Butler, used on the Post-Round Screen when the user has failed the scenario (any condition other than pass). The eyebrow on the monocle side is raised, the monocle on that side is slightly tilted, and an optional subtle forehead wrinkle on the raised side may be rendered. All three Butler assets share identity markers: golden monocle, sharp mustache, visible eyebrows, faceless otherwise (no eyes or mouth rendered). Frog dealer assets (one base PNG with two transparent expression overlay PNGs): frog_base.png: front-facing view of the Frog with no expression. The Frog is always rendered front-facing in this application; there is no side-profile Frog variant. frog_expression_pass.png: transparent overlay PNG containing only two small light pink circles positioned to land on the Frog's cheeks when layered over frog_base.png. Used when the user has passed the scenario. frog_expression_fail.png: transparent overlay PNG containing only the Frog's protruding tongue, positioned to render downward in a small mocking expression when layered over frog_base.png. Used when the user has failed the scenario. The Frog wears the same golden monocle as the Butler. The Frog overlays render compositionally on top of frog_base.png at the same screen position; the renderer composites them with standard alpha blending. If a Frog expression overlay has not yet finished loading when the user reaches the Post-Round Screen, the base Frog renders without expression. When the overlay completes loading, it fades in over approximately 200ms on the next frame boundary, with no jarring snap-in. The Base Audio Engine: Runs background music in a non-blocking thread via miniaudio.h or Emscripten's HTML5 audio bridge. Within the user's selected genre, the engine plays all currently-unlocked tracks in randomized order with approximately 3-second crossfades between tracks. Once all unlocked tracks have played in the current shuffle cycle, the queue automatically reshuffles and continues. This prevents single-track loop fatigue during long sessions. When a user purchases a new track in the Shop, the track is automatically added to the active shuffle pool for its genre and will appear in subsequent cycles. Audio playback honors all granular controls in the Audio Settings: Mute all, Mute music (independent of SFX mute), Mute SFX (independent of music mute), and the volume slider with paired text input. Audio autoplay is blocked by most browsers until the user has interacted with the page. Music playback is therefore initiated on the first user gesture (any click) on the Root screen, not on Root render. Until the first user click, no music plays. After the first click, music begins normally and continues across screen transitions for the remainder of the session. If the user removes the last track from the active genre's shuffle pool during playback (via the Shop's Remove action), music halts immediately on the next frame — no fade-out, no completion of the currently-playing track. If the user adds a track to the empty pool, playback resumes from the next track in the new shuffle order. The SFX Engine: Plays short sound effect samples in response to specific game events. Implemented via miniaudio.h alongside the Base Audio Engine, sharing the same audio thread. SFX samples are bundled with the initial WebAssembly package (estimated under 3MB total). The SFX library includes: Card Deal: Plays once when a scenario spawns. Single sound for the entire deal (hole + community), not per-card. Button Click Confirmation: Plays when the user presses 1-6 to focus a math input box. Subtle, fast, confirms keyboard input registered. Especially valuable when HUD is hidden. Chip Push: Plays alongside the chip slide animation in Caller scenarios when chips animate from the opponent's seat to their forward-pushed position. Side Pot Split: Plays when an All-In side pot scenario triggers and chips visualize splitting into the two pots. Modal Open Swoosh: Plays when Settings, Help, Shop, or any confirmation modal opens. Modal Close Swoosh: Plays when Settings, Help, Shop, or any confirmation modal closes. Frog Easter Egg Trigger: Plays once when 22 consecutive dealer clicks complete the Butler ↔ Frog toggle. Tonally distinct (a small 'bloop' or amphibian-coded sound). Slide In: Plays during the Game → Post-Round 350ms slide transition. Plays at scenario submission moment as the screen content slides right-to-left. Slide Out: Plays during the Post-Round → Game 350ms slide transition. Plays when the user commits the Again button as the screen content slides left-to-right. Slide In and Slide Out are designed as a complementary pair — sourced or generated to feel tonally connected, representing the recap-loop rhythm. Sourcing approach: a single source SFX may be used with the second instance played in reverse, or two separately-sourced complementary SFX may be used. Determined during asset acquisition. Audio Choreography at Scenario Spawn: To prevent simultaneous SFX from competing for the user's attention, scenario-spawn audio is sequenced rather than simultaneous. The Card Deal SFX plays at 100% volume at T=0ms. For Caller scenarios and side pot scenarios involving chip movement, the Chip Push SFX (or Side Pot Split SFX, as appropriate) plays at 70% volume at T=400ms, after the Card Deal SFX completes, synchronized with the chip slide animation. This sequencing mirrors the natural rhythm of poker play (cards before bets) and creates a clean audio-visual moment at scenario start. Audio on Post-Round Screen entry consists only of the Slide In SFX accompanying the screen transition. The dealer fade-in, modal fade-in, and pass/fail expression rendering are all silent. There is no chime, no correct/incorrect audio cue, no dealer voice line — the architecture deliberately omits performance-feedback audio to avoid conditioning the user toward audio-based grading rather than self-evaluation through the modal contents. SFX Volume Hierarchy: All SFX play through the global Volume slider in Audio Settings. The Mute SFX toggle silences all SFX while leaving music unaffected. The Mute Music toggle silences music while leaving SFX unaffected. The Mute All toggle silences everything. Asset Sourcing: SFX samples are sourced from royalty-free libraries (freesound.org, Zapsplat, Pixabay, etc.) under permissive licenses. Each sample is 1-3 seconds long, with cumulative file size for all 9 samples estimated under 3MB. Music tracks are sourced from royalty-free libraries (Free Music Archive, Pixabay Music, Incompetech, etc.) under permissive licenses (CC0 or CC-BY). Tracks requiring attribution under CC-BY are credited in the About / Credits modal in Settings. Tracks are transcoded to ~3MB MP3 (96-128kbps) during asset preparation to meet the per-track size budget specified in Module 7's Lazy-Loader.

## Module 3: The Emscripten Bridge (The Wasm Layer)

The Main Loop: emscripten_set_main_loop drives the ImGui rendering cycle at the browser's natural refresh rate via requestAnimationFrame (typically 60Hz on standard displays, up to 144Hz on high-refresh-rate monitors, and adaptive on low-end devices). All animations are timed in milliseconds rather than frames so they complete at the same wall-clock duration regardless of frame rate. The Reduce Motion setting, Background atmospheric movement toggle, and Particle drift toggle in Settings provide additional opt-out levers for users on low-end devices. Loading Strategy: To minimize initial perceived load time, asset loading is tiered by when assets are actually needed.

### Tier 1 (Synchronous Initial Load — required before app renders)

The compiled WebAssembly binary, background room PNG (heavily-blurred Root variant), the 4 main UI button assets for the Root screen, the Home icon, the dealer button asset, and the front-facing Butler dealer assets (butler_neutral.png and butler_raised.png — promoted to Tier 1 to eliminate any race condition where the user could finish a scenario and reach the Post-Round Screen before the front-facing Butler has loaded). Estimated 2-4 MB compressed. The Root screen renders as soon as Tier 1 completes. The loading screen displays during Tier 1 download with the following composition: the heavily-blurred Root background renders behind the dealer button, which is positioned at the center of the screen at large scale (approximately 150-200 pixels diameter). As Tier 1 assets download, a white arc traces clockwise along the dealer button's inset dashed ring, filling proportionally to download progress. When the arc completes a full 360° revolution along the dashed ring, Tier 1 has finished loading and the screen transitions to the Root screen. The dealer button does not rotate during loading; the white-arc fill is the only motion. There is no separate progress bar — the dashed ring fill IS the progress indicator. After Tier 1 completes, the dealer button on the table during gameplay renders without the white arc (the dashed ring reverts to purely decorative). Tier 1 failure handling: If any Tier 1 asset request fails (404, network timeout, CORS error, or any other fetch error), the system retries the failed request up to three times with exponential backoff (immediate, 2 seconds, 10 seconds). If all three attempts fail, the loading screen transitions to a dedicated error screen. The error screen displays a centered message: "Couldn't load. Check your connection and try again." Below the message, a single button labeled "Retry" using bg_button_default / bg_button_hover / bg_button_active and text_button. Clicking the Retry button reloads the page entirely (window.location.reload), restarting Tier 1 from scratch. The error screen uses bg_primary as the background; no other elements render. The loading screen's dashed-ring progress indicator is not visible on the error screen.

### Tier 2 (Background Load — fetched asynchronously after Root screen renders)

the side-profile Butler PNG (butler_profile.png), poker table PNG, all 52 card PNGs, chip denomination textures, all-in markers, the Exit door glyph, the Copy and Share glyphs used on the Post-Round Screen, the side-pot stacked-chip icon, and the Modal Open/Close Swoosh SFX (since user may click Settings/Shop/Help while on Root). Estimated 4-6 MB. Loads in the background while the user is on the Root screen. Does not block any user interaction; if the user clicks Play before Tier 2 completes, the system finishes loading needed Tier 2 assets and then transitions. Tier 2 failure handling: If any Tier 2 asset request fails, the system retries up to three times with exponential backoff (immediate, 2 seconds, 10 seconds). If all three attempts fail, the failed asset is marked as unavailable in the asset registry. If the user later attempts a navigation or interaction that requires that asset (e.g., clicking Play when the table or card PNGs failed), the navigation is blocked and an error screen displays before the navigation completes. The error screen uses the same visual treatment as the Tier 1 error screen: centered message "Couldn't load. Check your connection and try again." with a Retry button that reloads the page.

### Tier 3 (On-Click Load — fetched when user navigates from Root to Mode Selection)

The remaining SFX samples (Card Deal, Button Click Confirmation, Chip Push, Side Pot Split, Frog Easter Egg, Slide In, Slide Out) and the default Lounge Jazz starter music track. Estimated 4-5 MB. Loads during the brief Mode Selection screen interaction. Tier 3 failure handling: If any Tier 3 asset request fails (SFX samples or default music track), the system retries up to three times with exponential backoff (immediate, 2 seconds, 10 seconds). If all three attempts fail, the failed asset is silently skipped — the corresponding SFX simply does not play when its triggering event fires, or the default music track does not play. The application continues to function without user-facing error indication.

### Tier 4 (On-Demand Load — fetched only when user takes a triggering action)

The Frog dealer assets — frog_base.png, frog_expression_pass.png, and frog_expression_fail.png — loaded on the first dealer click within a session that triggers the Butler ↔ Frog toggle. All three Frog assets are fetched together as a bundle since they are interdependent (the overlays only make visual sense when composited on the base). The 3 alternate genre starter tracks (Classical, Bossa Nova, Ambient — loaded when their genre becomes the active selection in Audio Settings, since they're pre-owned and pre-in-shuffle but not yet cached locally) and the 8 purchasable music tracks (already specified as lazy-loaded via the Lazy-Loader in Module 7). User never waits for Tier 4 loads to begin interacting with the app. Tier 4 failure handling: If any Tier 4 asset request fails (Frog dealer assets, alternate genre starter tracks, or purchased music tracks), the system retries up to three times with exponential backoff (immediate, 2 seconds, 10 seconds). If all three attempts fail, the failed asset is silently skipped. For Frog assets, this means the dealer continues to render as Butler even after the user has triggered the easter egg (the click count is satisfied but the visual swap cannot complete). For music tracks, the track is skipped in the shuffle pool per the existing Lazy-Loader behavior. Users can trigger a manual retry by toggling the track's shuffle pool state in the Shop (Remove then Add) per the existing Module 7 spec; for Frog assets, manual retry occurs on the next session start. Frog Asset Loading Race: If the user enters Frog mode and immediately completes a scenario before all three Frog assets finish loading, the Post-Round Screen renders whichever assets are available. If frog_base.png is loaded but the expression overlays are not, the base Frog renders alone with no expression. When the relevant expression overlay completes loading, it fades in over approximately 200ms on the next frame boundary, as specified in Module 2. Compression: All static assets are served gzip- or brotli-compressed. The WebAssembly binary uses Emscripten's built-in compression flags. PNG assets are optimized with TinyPNG-equivalent tooling before bundling. CDN Delivery: Static assets (PNGs, music tracks, SFX, WebAssembly binary) are served from a CDN (Cloudflare or AWS CloudFront) for edge-cached global delivery, reducing effective latency for geographically distant users. Service Worker Caching: After the first visit, the Service Worker caches all assets that have been fetched. Subsequent visits load from browser cache, reducing time-to-interactive to under 1 second for returning users. The cache is versioned so app updates invalidate stale assets correctly. Streaming Audio: Music tracks play via HTML5 audio streaming. Playback begins as soon as enough bytes are received rather than waiting for full download completion. This applies to Tier 3 and Tier 4 music tracks.

## Module 5: The Math Interrogator (The Dynamic Drill Overlay)

Layout: Math input boxes are rendered along the left-middle of the Game screen, vertically centered and spreading up/down equally if the scenario branch produces many inputs. Each input box is labeled with its math name (e.g., 'Pot Odds', 'Outs', 'Equity', 'EV', 'Fold Probability'). Boxes are always visible during gameplay regardless of scenario type — boxes that aren't relevant to the current scenario are hidden, leaving only the inputs the user must answer. Input Format (Numeric Boxes): Boxes accept numeric input only — digits, a single decimal point, and an optional leading minus sign for EV. No percentage symbols, no commas, no other characters. The user types percentage values as raw numbers (e.g., '30' represents 30%, not 0.30). This is intentional: the trainer is practicing math thinking, not formatting. Visual State (Numeric Boxes): An unfocused empty box renders with its standard outline. An unfocused filled box renders with the user's typed value inside. A focused box renders with an additional white outline overlaid on the standard outline. Bet Size Input Mode: Bet Size is rendered as a focus-grouped row of four buttons (1/3 Pot, 1/2 Pot, Full Pot, Overbet) alongside the math input boxes, NOT as a numeric text input. Generating the precise optimal bet amount is too cognitively expensive on top of the other math; tier selection is the appropriate granularity. The user Tabs into the bet size group as a single tab stop. While the group is focused (visually indicated by a 2px outline using the border_focus token, rendered around the bounding box of the entire row of four buttons rather than around any individual button — see Notes — Keyboard Focus Behavior for the focus indicator visual spec): Keys 1-4 select the corresponding button in visual order (1=1/3 Pot, 2=1/2 Pot, 3=Full Pot, 4=Overbet) Global keybinds for math input box focus (1-6 per Settings) are suppressed while bet size has focus Tab moves focus to the next input box and does NOT cycle within the bet size group Shift-Tab moves focus to the previous input box and does NOT cycle within the bet size group The selected bet size persists across the scenario and is visually highlighted with the accent color. If the user Tabs through the bet size group without selecting, the bet size remains unselected and grades as wrong on submission. Mouse click on any of the four buttons also selects it, with identical visual feedback. The keyboard scope restores to global 1-6 keybinds as soon as the user Tabs or Shift-Tabs out of the bet size group. Keybinds: Per the Keyboard Shortcuts in Settings — 1-6 focus the corresponding input box (suppressed while bet size group is focused), Tab cycles to next input, Shift-Tab cycles to previous input, Enter submits all answers. Backspace clears characters within the focused numeric box. Keybinds train the user toward keyboard-driven workflow given strict time pressure constraints. Arrow keys within a focused math input box do not increment or decrement the value. Math input boxes accept unbounded decimal values (positive and negative for EV, positive integers for Outs, positive decimals for probabilities), and arrow-key increment/decrement is not meaningful at that scale. Arrow keys are reserved for bounded integer inputs (settings sliders, custom mode percentages, volume) and for navigating within grouped focus elements (tier tab strip, range slider handles). See Notes — Keyboard Focus Behavior. The Context-Aware Modal: Input elements spawn dynamically based on the scenario's branch: Caller Scenarios: Spawns [Pot Odds], [Outs], [Equity], [EV] — all numeric input boxes. Aggressor Pure Bluff or Value Bet: Spawns [Fold Probability %], [Bet Size buttons], [EV]. Aggressor Semi-Bluff: Spawns [Fold Probability %], [Equity if Called], [Bet Size buttons], [EV]. Multi-Tier Bet Sizing Behavior: When the Bet Sizing Engine setting is on (default), Aggressor scenarios present the same scenario at multiple bet sizes. Inputs that are bet-size-dependent (Fold Probability, EV, Bet Size selection) reappear at each tier for the user to recompute, since the answer differs per tier. Inputs that are bet-size-independent (Equity if Called, Outs) lock after tier 1 and do NOT reappear in subsequent tiers — the user does not re-answer something whose value does not change with bet size. The user's tier-1 answer for these inputs persists and is reused for grading at each subsequent tier. Submission: Pressing Enter submits all answers simultaneously across all visible inputs. Any unfilled numeric boxes are graded as incorrect. An unselected Bet Size group is graded as incorrect. There is no partial credit, no method-vs-result evaluation — wrong is wrong. The system does not display correct work, partial reasoning, or hints. The trainer is not a math test review tool; it is a calibration tool. During an active tutorial, Enter is suppressed until all visible inputs across all tiers are filled; once all inputs contain values, Enter submits normally. Tab remains the standard advance mechanism between inputs in both tutorial and gameplay. See Notes — Tutorial System for full tutorial behavior. The Post-Round Grader: Applies strict Margins of Error to each input: Dollar EV: ±5% relative variance, with a $0.50 absolute floor (variances smaller than $0.50 in absolute terms are treated as $0.50 to prevent unfair grading on tiny EV scenarios). Probabilities (Pot Odds, Equity, P(fold)): ±5 percentage points absolute. Bet Size: Exact button selection match required (the user picked the correct sizing tier or did not). Outs: Exact integer match required. The Post-Round Screen: After Enter submission on the Game screen, the application transitions to the dedicated Post-Round Screen via a 350ms slide animation (right-to-left) accompanied by the Slide In SFX. The Post-Round Screen displays the user's per-input grading results in a centered stat modal, time grade, scenario-level summary tab for multi-tier Aggressor scenarios, and forward navigation via the Again button (double-confirmation), Exit door glyph (returns to Mode Selection), or persistent cluster Home icon (returns via ceremonial fade-to-black-and-back transition). Tomatoes are explicitly NOT displayed on the Post-Round Screen per Module 7's currency-display scoping rule. The full specification including layout, dealer rendering, performance expressions, stat modal contents, tier tabs, Summary tab, Again button mechanics, Scenario ID utility buttons, and transition behavior is documented in the 'Notes — Post-Round Screen' section. Module 5 is responsible for: (1) generating the per-input grading results consumed by the Post-Round stat modal, (2) determining the pass/fail expression state passed to the dealer asset rendering layer per the tomato-award threshold, and (3) preserving the bet-size-independent input answers (Equity if Called, Outs) from tier 1 of multi-tier Aggressor scenarios so they can be echoed at every tier tab in the recap.

## Module 6: The Temporal Layer (The Ghost Timer)

The Delta Timer: Tracks time elapsed since the scenario spawned. The timer pauses when any modal opens over the Game screen (Shop, Help, Settings, or any other modal overlay) and resumes when the modal closes. While paused, the Visual Countdown timer freezes at its current displayed value. This allows users to consult reference material, adjust settings, or shop without losing time on the active scenario. The Visual Countdown: Renders a small numeric timer in the top-right region of the Game screen, directly below the persistent cluster, right-aligned to the cluster's right edge. Format is integer seconds followed by the letter 's' (e.g., '18s', '5s', '0s'). Plain text, no icon, no box, no border. Smaller than the persistent cluster icons in visual weight. Hidden by default; user enables via Gameplay Settings. The Overtime Tracker: If elapsed time exceeds the target time, the scenario does not fail. The timer continues from '0s' and counts upward ('1s', '2s', '3s', etc.), with the text color switching from text_secondary to state_fail (red) for the entire overtime duration. There is no plus sign prefix; the red color is the sole overtime signal. Default time values scale by street and scenario type: Pre-flop Aggressor/Caller: 10s; Flop: 18s; Turn: 22s; River: 22s. Add 50% to base time if multi-tier bet sizing is on. Add 25% to base time if the scenario is a side pot.

## Module 7: The Retention Engine (The Shop)

The Currency State: The in-app currency is named 'tomatoes' and represented visually by a stylized tomato icon (sourced via Module 2 Texture Loader). Tomatoes operate in two parallel tracks. Spendable Tomatoes is the amount the user can spend in the Shop on track unlocks. Lifetime Tomatoes is a cumulative total that never decreases when tomatoes are spent. Both values increase together when tomatoes are awarded; only Spendable Tomatoes decreases when the user purchases something. Tomatoes are awarded only if two conditions are met: (1) All math inputs pass the accuracy check, AND (2) The total scenario time is at or under the self-set target time. Tomatoes are displayed on exactly three surfaces and nowhere else: (a) the Shop modal (Spendable Tomatoes shown in the top-right of the modal alongside item prices), (b) the user's profile/account view (both Spendable and Lifetime Tomatoes displayed), and (c) the global Leaderboard view (Lifetime Tomatoes shown as the ranking metric). The HUD during gameplay does NOT display tomatoes. The Post-Round Screen does NOT display tomatoes or any currency feedback — neither the stat modal, the dealer area, the Scenario ID block, nor any other element on the Post-Round Screen renders tomato counts or award notifications. This separation keeps active-training surfaces focused on math and time calibration without currency cognitive load. Tomatoes only become visible when the user navigates to one of the three meta surfaces. The dual-track model means spending in the Shop never penalizes the user's leaderboard standing. Account creation migration: When a guest user creates an account, three pieces of state migrate from IDBFS to server-side: Current Spendable Tomatoes balance Lifetime Tomatoes total Full list of tracks unlocked via Shop purchases All three upload as the initial account state. After migration, the server is authoritative for all three. The user's unlocked tracks remain unlocked; their wallet and lifetime totals continue accumulating from their pre-account values. The Persistence Layer (IDBFS + Account State): Emscripten's IndexedDB file system stores the user's currency wallet, unlocked track list, shuffle pool composition, mode preferences, and the has_seen_tutorial_prompt flag (see Notes — Tutorial System for full lifecycle) in the browser's local storage. For logged-in users, this state is also synced to server-side account state keyed by the user's Auth0 account ID. On session start, the server state is the source of truth: IDBFS reconciles to match. On state change (tomato earn, Shop purchase, Add/Remove shuffle action), IDBFS writes immediately and a background sync writes to the server. On sign-in from a guest session, the local IDBFS state migrates to the server as the initial account state (per the migration rules in the Account State and Tomato Persistence subsection). On sign-out, IDBFS state remains intact for offline use as a guest. The Lazy-Loader: When a track is unlocked via Shop purchase OR auto-loaded from the user's previously-unlocked list at session start, the engine queues the track for download via asynchronous fetch (emscripten_async_wget). Downloads process serially — the loader handles one download at a time and processes the next when the current completes. This prevents bandwidth saturation when a user purchases multiple tracks rapidly. Purchase commits are decoupled from download completion: a Buy action commits instantly (Spendable Tomatoes decrements, the track is marked owned in IDBFS), and the audio download proceeds in the background. The user's purchase does not block on the download; ownership is immediate. Tracks play via HTML5 audio streaming (per Module 3) — the audio element references the track URL, and the browser handles buffering as playback proceeds. The engine does not retain full decoded audio data in memory; the active audio element streams from cached or fetched bytes as needed. Failed downloads: if a fetch fails (network error, timeout), the track remains owned but is flagged as "not yet locally available" in IDBFS. The engine retries the download on next session start. If the track is in the active shuffle pool when its turn comes during playback and is still not available, the engine skips it (advances to the next track in the pool) rather than blocking playback. Users can trigger a manual retry by toggling the track's shuffle pool state (Remove then Add) in the Shop. The Shop UI: A centered modal overlay opened from the persistent cluster's Shop icon. Fixed dimensions, no scrolling within the modal — all 12 tracks render visible at once. Modal header: The top-right of the modal contains, from left to right: the Leaderboard icon (per the Leaderboard Surface spec), the Spendable Tomatoes count alongside the tomato icon, and the modal's X close button. Clicking the Leaderboard icon swaps the modal content to the Leaderboard view (per the Leaderboard Surface spec). The X close button dismisses the entire modal and returns to the underlying screen. Layout — sections by genre: The modal body renders all 12 tracks grouped into four genre sections stacked vertically: Lounge Jazz, Classical, Bossa Nova, Ambient. Each section has a small genre header label and three track rows below it. No tabs, no scrolling — every track is visible on first open. Track row layout: Each row spans the full modal width with three positional zones: Far left: a small filled circle indicator (accent_primary). Present only when the track is currently in the shuffle pool for its genre. Absent when the track is locked or owned-but-not-in-rotation. The leftmost column of dots gives the user an at-a-glance view of their entire rotation across all genres. Left of center: track name in text_primary. Right side: the action button. Immediately to the left of the button (outside it, not inside): the price displayed as the tomato icon followed by a number, present only when the track is in the Locked state. Owned tracks show no price element — just the button on the right. Action button states: Each track row's button has exactly one of three states based on the track's status: Locked — Button label: "BUY". Two-click armed-state confirmation: first click arms the button (label changes to "CONFIRM," background switches to bg_button_armed), second click commits the purchase. On commit: Spendable Tomatoes decrements by the price, the track transitions to Owned-not-in-shuffle state, and the row updates to show the Add button. If the user has insufficient Spendable Tomatoes, the price renders in state_fail (red), the button is disabled (non-interactive, rendered with reduced opacity), and clicks do not arm. Owned, not in shuffle — Button label: "ADD". Single click adds the track to the genre's shuffle pool via the Base Audio Engine. The row updates: the leftmost dot indicator appears, and the button transitions to the Remove state. No confirmation pattern — single click commits. Owned, in shuffle — Button label: "REMOVE". Single click removes the track from the genre's shuffle pool. The row updates: the leftmost dot indicator disappears, and the button transitions to the Add state. Default tracks: The first track of each genre is permanently owned and pre-added to that genre's shuffle pool on first session. These default tracks render in the Shop alongside paid tracks, in the Owned-and-in-shuffle state. They can be removed from the shuffle pool like any other owned track, but they cannot be re-locked — their state machine has only two states (in-shuffle ↔ out-of-shuffle), no Buy state. The Shop renders all 12 tracks with no visual distinction between "free starter" and "purchased" tracks; both are simply "owned." Shuffle pool empty state: If a genre's shuffle pool is empty when that genre is the active selection in Audio Settings, no music plays until the user adds at least one track. There is no warning or fallback — silence is the explicit state. Persistence: Two pieces of state persist independently via IDBFS (and via account state for logged-in users): the owned-tracks list, and the per-genre shuffle pool composition. Purchasing changes the owned list. Add/Remove changes the shuffle pool composition. The two are decoupled. All Shop actions (Buy commit, Add, Remove) autosave on action — no Save button. Writes go to IDBFS immediately and to server-side account state in the background for logged-in users. Modal close: X close button in top-right, click-outside dismissal, or Escape key — all dismiss the modal and return to the underlying screen. The Music Shop (Expanded): The Shop sells music tracks across four instrumental genres (Lounge Jazz, Classical, Bossa Nova, Ambient), 3 tracks per genre, totaling a library of 12 tracks. The genre dropdown in Audio Settings selects which genre's shuffle pool is currently playing; track-level interactions (purchase, add to shuffle, remove from shuffle) happen exclusively in the Shop UI. When a user adds a newly-purchased or previously-owned track to a genre's shuffle pool, the track joins that genre's current shuffle cycle without requiring an audio engine restart — it appears in the next reshuffle (or sooner if the engine implements live queue updates). Each paid track costs 25 tomatoes. Total catalog cost to unlock all 8 paid tracks is 200 tomatoes. The Leaderboard Surface (Behavior): The leaderboard icon is placed in the top-right of the Shop modal, immediately to the left of the modal's X close button (per the Shop UI spec). Clicking the leaderboard icon opens the leaderboard view (modal swap — the Shop modal's content is replaced by the leaderboard content within the same modal frame). The leaderboard displays the top 100 globally ranked accounts by Lifetime Tomatoes (cumulative, never decreased by Shop spending). The leaderboard refreshes daily at 00:00 PST. The leaderboard is opt-in: users must enable leaderboard visibility in the Tomatoes settings to appear on the public leaderboard. Users who have not opted in still see the leaderboard and can search it, but do not appear on it themselves. No anti-cheat measures are implemented. Users found gaming the leaderboard mechanic are not policed by the system. This is a deliberate design choice, documented in the Terms of Service. The Leaderboard View (UI): Modal frame: The leaderboard view replaces the Shop modal's content via in-place content swap. The modal frame itself (size, position, X close button location) does not change. Modal header (top-left): Per the Modal Popup Conventions, a top-left header renders. The icon is a trophy or similar leaderboard glyph in text_primary, followed by the rounded pill containing the title "Leaderboard". Immediately to the right of the pill, in smaller text using text_secondary, the static text "(refreshes daily at 00:00 PST)" renders inline. Modal header (top-right): The top-right of the modal contains, from left to right: the Shop icon (the same storefront icon used in the persistent cluster), the user's Lifetime Tomatoes count alongside the tomato icon, and the modal's X close button. Clicking the Shop icon swaps the modal content back to the Shop view (the inverse of the Leaderboard icon's swap behavior in the Shop). The X close button dismisses the entire modal and returns to the underlying screen (Mode Selection, Game, or Post-Round, depending on where the modal was opened from). The Lifetime Tomatoes display in this position mirrors the Shop modal's Spendable Tomatoes display, contextually showing the metric most relevant to the active modal view: Spendable in the Shop (where it's spent), Lifetime in the Leaderboard (where it's the ranking metric). Search bar: A horizontal search input sits below the modal header, full-width minus modal padding. Placeholder text reads "Search leaderboard" in text_placeholder. The input accepts up to 32 characters; additional input beyond 32 characters is rejected at the keystroke level. The input accepts unicode characters (including non-ASCII letters, kana, emoji, etc.). Special characters including punctuation, spaces, and symbols are treated as literal characters in the substring match; no regex parsing is performed. As the user types, the displayed list filters live to show only entries whose username contains the typed substring (case-insensitive plain-substring match). Whitespace in the search input is treated as literal characters (typing only spaces filters for usernames containing that many consecutive spaces; typing then deleting back to an empty input restores the full top-100 list). When the user presses Enter, the filter clears (full top-100 list is restored) and the view scrolls to the highest-ranked match, with that row highlighted via a row-background tint using accent_secondary. If multiple matches exist, only the highest-ranked match is highlighted on Enter. Leaderboard list: A vertically scrolling list of up to 100 rows. Each row contains: Far left: rank number (1, 2, 3...) in text_primary. Left of center: username in text_primary. Right side: the lifetime tomato count followed by the tomato icon, both right-aligned to the row's right edge. Hovering over a row tints the entire row's background with bg_button_hover. The hover tint extends edge-to-edge across the row. The currently-signed-in user's row, if present in the top 100, is permanently highlighted with a subtle background tint using accent_secondary at reduced opacity (~30%) to distinguish it from hover. Persistent "your rank" row: A row pinned to the bottom of the leaderboard view, separated from the scrollable list by a small grey gap (using separator at slightly higher opacity than the in-list separators). This row is always visible regardless of scroll position. Same accent_secondary background tint as their in-list row if they are in the top 100, otherwise default row background. Logged-in user with leaderboard opt-in disabled: The row renders with the user's actual rank (server-side computed) and their username and tomato count, but appended to the right of the username, in text_secondary smaller text: "(opted out — opt in to appear on leaderboard)". Guest user (not signed in): The row contains no rank number, no tomato count. Instead it displays a centered message in text_secondary: "Sign in or sign up to see your rank." Empty / loading states: If the leaderboard data is still loading from the server, the list area renders a loading indicator (centered, text reading "Loading leaderboard..."). On retry failure, the error message and Retry button render again, allowing repeated retries.
